security policy

Disclose before publishing.

Asclepius is a trust substrate. A vulnerability in this code is a vulnerability in every clinical-AI deployment that uses it. We treat reports the way the project asks others to: with cryptographic care, written promptly, fixed at the root.

How to report.

Send the report to security@asclepius.health. Encrypt with the PGP key below if the finding involves an active exploit. Include reproduction steps, affected versions, and any artifacts (a tampered ledger, a malformed bundle, a regex bypass corpus, etc.).

If you are not sure whether something is in scope, send it anyway — we'd rather triage a non-issue than miss one.

Scope.

In scope:

  • Cryptographic flaws: chain forgery, signature bypass, hash collisions, key-derivation issues.
  • Memory safety: buffer overflow, use-after-free, double-free in the C++ core.
  • Injection: SQL injection in asclepius-cli, prompt-injection that escapes the policy chain.
  • Verification bypass: a tampered bundle that verify_bundle() accepts.
  • Consent / scope bypass: an inference that reaches the model despite missing or revoked consent.
  • Side channels: PHI leakage through ledger metadata, drift histograms, or evidence bundles.

Out of scope:

  • The wrapped model's outputs (that is what the policy chain is for; report a missing policy as an enhancement).
  • Operational misconfigurations on a customer deployment (unless the project documentation explicitly recommended the misconfiguration).
  • DOS via legitimate ledger growth — write a benchmark, not a CVE.
  • Theoretical post-quantum risk (we know; v1.0 ships ML-DSA migration).

Supported versions.

We patch the latest minor and the immediately previous minor. Earlier versions get advisories but no backports.

Project signing keys.

Released artifacts are signed by the project's release key. Evidence bundles emitted by individual deployments are signed by their own ledger keys; the project does not co-sign customer bundles.

What to expect.

  • Acknowledgement within 48 business hours.
  • Triage within 5 business days. We assign a severity (informational / low / medium / high / critical) and an estimated fix window.
  • Fix shipped to main + a patch release. Critical issues ship within 14 days.
  • Coordinated disclosure: 90 days from triage, or 7 days after the patch release — whichever is sooner. We will work with you to extend the window if there is responsible reason to.
  • Credit: with your permission, in the changelog and the release advisory. Anonymous reports are honoured.
