§01 · seven runbooks · oncall ready

What to do at 3 a.m.

The substrate is reliable. The world it runs in is not. Below are seven runbooks for the incidents an operator will face — written for the person whose pager is on, in the language they will read at 3 a.m. Each follows the same shape: detection · containment · investigation · remediation · postmortem.

PB-01 ledger truncation detected P1 PB-02 scheduled key rotation P3 PB-03 signing key compromise P1 PB-04 drift threshold exceeded P2 PB-05 consent revocation cascade P2 PB-06 evidence-bundle subpoena P2 PB-07 sidecar network partition P2
PB-01 P1 · 30-min ack owner · ledger oncall

Ledger truncation detected.

The verifier reports a gap in seq or a prev_hash mismatch. Treat as adversarial until proven otherwise. The substrate is designed so this is the loudest possible alarm — gaps are forbidden by spec.

detect

  • verifier alert: chain.broken
  • watchdog: nightly verify_chain --strict failed
  • auditor opens compare.html · sees amber row

contain

  • flip runtime.read_only=true — no new appends
  • snapshot the SQLite WAL before investigating
  • page security oncall · do not reboot

investigate

  • asclepius ledger inspect --range last:1000
  • diff against last published merkle root in changelog feed
  • check WAL checkpoint + signing-key audit log

remediate

  • restore from last verified snapshot
  • replay from a downstream subscriber if available
  • publish operator advisory in changelog.html
# freeze writes, snapshot, verify range
asclepius admin set runtime.read_only=true
asclepius ledger snapshot --out /var/asc/snap-$(date +%s).db
asclepius ledger verify --range last:1000 --strict
# compare to last published root
curl https://asclepius.health/feed.json | jq '.items[0].merkle_root'
Postmortem requirements. Within 7 days of resolution, publish a postmortem to changelog.html with: timeline, root cause, affected encounters (by seq range), customer notification log, and which auditing party caught it first. If the answer is "Asclepius' own watchdog," good. If the answer is "a customer," that goes in the quarterly board report.
PB-02 P3 · scheduled owner · platform

Scheduled key rotation.

Ed25519 ledger keys rotate every 180 days. Rotation is a planned, signed event — not an emergency. The new key takes over after a single key.rotated entry chains the two together. The chain survives the rotation; the verifier never sees a discontinuity.

prepare

  • announce in change calendar · 14-day window
  • generate new key in HSM (never on disk)
  • countersign new pubkey with previous private key

execute

  • asclepius keys rotate --key-id KID-2
  • writes key.rotated entry · signed by both keys
  • publish new pubkey + sig in humans.txt

verify

  • download fresh evidence bundle · verify with new pubkey
  • verify key.rotated entry with old + new sigs
  • confirm changelog entry posts a new root

retire

  • archive old privkey to cold custody (not delete)
  • 30-day overlap before old key is fully revoked
  • downstream verifiers must accept both during overlap
# end-to-end rotation, dry-run first
asclepius keys rotate --key-id KID-2 --dry-run
asclepius keys rotate --key-id KID-2 --confirm
asclepius ledger verify --range last:50
# publish new pubkey block
asclepius keys export --pem KID-2 > site/assets/pubkey-KID-2.pem
Common pitfall. Operators sometimes try to rotate by writing the new pubkey into config and restarting. That breaks chain continuity and forces every downstream verifier to re-trust manually. Always rotate via keys rotate, which produces the chained key.rotated entry the verifier expects.
PB-03 P1 · 15-min ack owner · security

Signing key compromise.

The privkey is suspected leaked, exfiltrated, or misused. Every entry signed with that key is now under suspicion. The substrate's job is to contain the blast radius: rotate, revoke, re-sign nothing, and let the forensic chain do the talking.

declare

  • page security oncall · file SEV-1
  • flip runtime.read_only=true globally
  • notify affected customers within 60 min

rotate

  • generate emergency key in air-gapped HSM
  • keys rotate --emergency writes a key.revoked entry
  • publish revocation to /.well-known/security.txt

scope

  • identify last-known-good seq before compromise
  • flag all entries between LKG and revocation as under review
  • do not delete · the chain is the forensic record

disclose

  • CVE filing within 72 h · security@asclepius.health
  • operator advisory in changelog
  • auditors re-pull bundles · re-verify with new pubkey
Why we never re-sign with the new key. Re-signing old entries with the new key would erase the very evidence that distinguishes pre-compromise from post-compromise records. The compromised entries stay signed by the compromised key. The chain itself is forensic. Lying about history would be the actual breach.
PB-04 P2 · 2-h ack owner · ml platform

Drift threshold exceeded.

Population Stability Index for one or more features crosses the configured threshold (typically PSI > 0.25). The model is making predictions on data that no longer matches its training distribution. This is not automatically a problem — but it requires a human to look.

detect

  • alert: drift.psi.breach per feature
  • which window (1d, 7d, 30d)?
  • which features · how many simultaneously?

classify

  • data-pipeline change → fix upstream, not the model
  • population shift → flag for retraining
  • seasonal → annotate; expect mean reversion

decide

  • downgrade model purpose? diagnostic → advisory_only
  • increase oversight? require attending sign-off threshold
  • halt model? rare; only when calibration also fails

document

  • append drift.acknowledged ledger entry
  • note decision + responsible owner
  • schedule re-evaluation · 30 days
# inspect drift snapshot for the breaching feature
asclepius drift show --feature "vitals.spo2" --window 7d
# diff distributions before/after
asclepius drift diff --baseline 2026-01-01 --current 2026-04-26
# acknowledge with rationale (becomes a signed entry)
asclepius drift ack --feature "vitals.spo2" --reason "sensor vendor change"
PB-05 P2 · 4-h ack owner · privacy

Consent revocation cascade.

A patient revokes consent for a purpose. Per the consent registry, future inferences for that patient stop immediately. Past inferences are not deleted — they are flagged in the bundle metadata so auditors know the patient's wishes have changed.

receive

  • consent-portal POST → consent.revoked entry
  • or operator-initiated via privacy ticket
  • SLA: revocation in effect within 24 h, target < 1 h

enforce

  • registry refuses subsequent consent.check
  • any in-flight inference completes; next is rejected
  • scope=revoked propagates to caches in < 60 s

annotate

  • past entries get consent.revoked_at metadata
  • bundle export marks affected rows in manifest
  • retention policy unchanged (deletion is not revocation)

confirm

  • send patient-facing confirmation (per HIPAA)
  • verify next inference attempt → consent.denied
  • log audit-trail in privacy-team bundle
Why we annotate, not delete. Deleting past entries would forge the record of what the model did and why. The compliant move is to keep history honest and let the consent revocation flow forward. If a regulator requests true deletion, that is a different process — see PB-06.
PB-06 P2 · 24-h ack owner · legal · platform

Evidence-bundle subpoena.

A court, regulator, or peer-review committee requests a bundle for an encounter, a date range, or a model version. The substrate is built for this. Export the bundle, verify it matches the request, hand it over with the public verifier instructions.

scope

  • read the request carefully · encounter? date range? model?
  • legal review for over-broad scope
  • no PHI beyond what is required

export

  • asclepius bundle export --encounter ENC-…
  • or --from 2026-01-01 --to 2026-03-31
  • output is USTAR tar · open with stock tar

verify

  • self-verify before delivery: verify_bundle output.tar
  • include the public verifier binary + pubkey
  • document the requesting authority + scope

deliver

  • hand off via legal channels · checksum receipt
  • operator log entry: bundle.exported
  • retain working copy in cold storage · 7 years
# export, self-verify, prepare delivery package
asclepius bundle export --encounter ENC-2026-01-15-r3 \
        --out /secure/2026-04-26-subpoena.tar
asclepius bundle verify /secure/2026-04-26-subpoena.tar
sha256sum /secure/2026-04-26-subpoena.tar  > delivery.sha256
# include the verifier so the recipient never has to trust us
cp build/tools/verify_bundle /secure/verifier.bin
PB-07 P2 · 1-h ack owner · sre

Sidecar network partition.

The runtime can no longer reach the central ledger backend. The local SQLite WAL keeps accepting writes — that is the design — but downstream subscribers will see a gap until reconnection. The substrate prefers local consistency over global immediacy.

detect

  • alert: backend.unreachable > 60 s
  • local WAL size growing; backpressure thresholds
  • verifier sees remote tail older than local tail

decide

  • continue local writes? (default · usually yes)
  • refuse new inferences? (only if WAL near disk limit)
  • page only if partition exceeds 30 min

reconcile

  • on reconnect, replay local WAL forward to backend
  • no merging — local seq is canonical
  • verify final root matches backend's expectation

monitor

  • watch tail divergence for 24 h post-incident
  • any retroactive edits → escalate to PB-01
  • capacity-plan WAL headroom for 4× the partition window

The runbook is part of the substrate.

Reliable systems are not just code that does the right thing under blue skies — they are the scripts you run when the sky is on fire. Every Asclepius release ships with these playbooks updated alongside the runtime. If a new failure mode is discovered in the field, a new playbook lands before the patch does.

A clinical-AI substrate that does not tell its operators how to fail is not a substrate; it is a liability disguised as a library.
