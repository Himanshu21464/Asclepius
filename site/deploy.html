Stand it up. Honestly.

The page the production-ops engineer reads first. Docker-compose for development, Postgres for the ledger backend in production, key-custody patterns, retention policy, multi-tenant isolation. Use these as a starting point and adapt to your existing stack.

Development.

One command. SQLite-backed ledger, fresh signing key generated on first run, key written to a sidecar file with mode 0600.

git clone https://github.com/Himanshu21464/Asclepius
cd asclepius

cmake -S . -B build -DCMAKE_BUILD_TYPE=RelWithDebInfo \
    -DCMAKE_POLICY_VERSION_MINIMUM=3.5
cmake --build build -j

./build/examples/01_wrap_scribe/example_wrap_scribe \
    /tmp/asclepius.db

To run against your own model, replace fake_scribe in examples/01_wrap_scribe/main.cpp with a callable that hits your real inference endpoint. The ledger structure does not change.

Production.

Two pieces: the runtime (linked into your service), and a Postgres-backed ledger. Below is a docker-compose that wires both together with the gRPC sidecar for non-C++ hosts.

# docker-compose.production.yml
version: '3.8'
services:
  asclepius-svc:
    image: asclepius/sidecar:0.1.0
    restart: unless-stopped
    environment:
      ASCLEPIUS_LEDGER_BACKEND: postgres
      ASCLEPIUS_LEDGER_DSN:     postgresql://asclepius:${PG_PASS}@db/asclepius
      ASCLEPIUS_KEY_PATH:       /var/lib/asclepius/keys/signing.key
      ASCLEPIUS_TENANT:         ${TENANT_ID}
    volumes:
      - keys:/var/lib/asclepius/keys:ro
      - bundles:/var/lib/asclepius/bundles
    ports:
      - "9090:9090"
    depends_on: [db]

  db:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_USER:     asclepius
      POSTGRES_DB:       asclepius
      POSTGRES_PASSWORD: ${PG_PASS}
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ['CMD', 'pg_isready', '-U', 'asclepius']
      interval: 10s

volumes:
  pgdata:
  keys:
  bundles:
Why Postgres. SQLite WAL is fine for single-node deployments and small ledgers (< 10M rows). Postgres lets you fan out reads, replicate to a second region for compliance, and run point-in-time recovery — all of which a regulator will eventually ask about. The ledger schema is identical; only the backend pointer changes.

Sidecar (gRPC).

Hosts that don't link libasclepius directly — Java, Go, .NET, internal-language apps — talk to the sidecar via gRPC. The wire format is in proto/asclepius/v1/sidecar.proto.

// Go example, using the generated stub
import (
    "context"
    pb "github.com/Himanshu21464/Asclepius-go/v1"
    "google.golang.org/grpc"
)

conn, _ := grpc.Dial("asclepius-svc:9090", grpc.WithInsecure())
client := pb.NewAsclepiusClient(conn)

resp, err := client.Run(ctx, &pb.RunRequest{
    Spec: &pb.InferenceSpec{
        ModelId:        "scribe@v3",
        Actor:          "clinician:dr.smith",
        Patient:        "pat:9f1a",
        Encounter:      "enc:01KQ…",
        Purpose:        "ambient_documentation",
        ConsentTokenId: tokenID,
    },
    Input: transcript,
})
if err != nil { return err }
soap := resp.Output  // already PHI-scrubbed, schema-validated, signed-and-committed

The sidecar emits identical ledger entries to the in-process runtime. A bundle exported from a sidecar deployment verifies against the same public key as one exported from in-process.

Key custody.

The Ed25519 signing key is the trust anchor. Three options, in increasing order of operational security:

  1. File-mode 0600. The default. Acceptable for a private VM that only the runtime user can read. Lose the file, lose your ability to verify past entries.
  2. Cloud KMS. Encrypt the key at rest with a KMS-issued envelope key (AWS KMS, GCP KMS, Azure Key Vault). The runtime decrypts on startup; key bytes never touch disk. Compromise the IAM role, compromise the key — same blast radius as the file approach.
  3. HSM (PKCS#11). The key never leaves the HSM. The runtime calls the HSM to sign each entry. ~5-10× slower per signature but unaffected by host compromise. Required for FedRAMP-High and most large IDN procurement.
Key rotation. Asclepius supports multiple key_ids in the same chain. Rotate by writing the new public key to the meta table and signing new entries with the new key. Verifiers learn the rotation point from the chain itself; old entries continue to verify against the old public key.

Retention.

The ledger is append-only by design. Two operational levers:

  • Off-site checkpointing. Pin the chain head to an external service on a regular cadence — a notary, a transparency log, a board-attested checkpointer. Asclepius does not implement this; it produces the artifacts (chain head + signature) that make external notarization cheap.
  • Cold-archive. Roll the ledger into a new database every quarter; archive the prior database alongside its public-key file. Each archive is independently verifiable.

If your jurisdiction requires deletion (GDPR right-to-erasure for non-clinical data), use a separate erasure-store for the actual content; the ledger holds only hashes by default, so erasing the content does not break the chain.

Multi-tenant.

One sidecar, one Postgres database, many tenants. Each ledger row carries a tenant field; the harness's export_bundle filters by tenant when producing per-tenant bundles. The signing key may be shared (cheap, one trust anchor for the whole platform) or per-tenant (expensive but more defensible if a tenant later leaves).

asclepius evidence bundle ./asclepius.db \
    --tenant kp-northwest \
    --window 30d \
    --out  evidence-kp-2026-04.tar

Monitoring.

The runtime emits Prometheus-shaped metrics on its admin port. Critical signals:

  • asclepius_inference_total{status,model} — wraps per second, by status (ok / blocked.input / blocked.output / model_error)
  • asclepius_inference_latency_seconds — histogram, target p99 < 1 ms
  • asclepius_drift_psi{feature,model} — current PSI; alert at > 0.25
  • asclepius_ledger_verify_seconds — verify duration on the most recent scheduled verify
  • asclepius_consent_denied_total — should be near-zero in steady state

Run a scheduled asclepius ledger verify every hour and alert on non-zero exit. A single mismatch is a paging-grade incident.

Scheduled evidence bundles.

The customer's compliance office wants a bundle on a schedule. A single cron line:

# /etc/cron.d/asclepius-bundles
0 2 * * 0  asclepius  asclepius evidence bundle /var/lib/asclepius/asclepius.db \
              --window 7d --out /var/lib/asclepius/bundles/$(date +\%F).tar \
              && sha256sum /var/lib/asclepius/bundles/$(date +\%F).tar \
                 > /var/lib/asclepius/bundles/$(date +\%F).tar.sha256

Mount that directory read-only into your cold-archive store; the SHA-256 file is the canonical artifact for a board-attestation memo.

Kubernetes.

For larger deployments, run the sidecar as a per-namespace Deployment with a Postgres StatefulSet (or external managed Postgres):

# asclepius-sidecar.yaml — Deployment + Service + ConfigMap
apiVersion: apps/v1
kind: Deployment
metadata: { name: asclepius-svc, namespace: clinical-ai }
spec:
  replicas: 3                # stateless; horizontal scale freely
  selector: { matchLabels: { app: asclepius-svc } }
  template:
    metadata: { labels: { app: asclepius-svc } }
    spec:
      serviceAccountName: asclepius
      containers:
      - name: svc
        image: asclepius/sidecar:0.1.0
        ports: [ { containerPort: 9090 } ]
        env:
        - { name: ASCLEPIUS_LEDGER_BACKEND, value: postgres }
        - { name: ASCLEPIUS_LEDGER_DSN, valueFrom: { secretKeyRef: { name: asclepius-pg, key: dsn } } }
        - { name: ASCLEPIUS_KEY_PATH,   value: /run/asclepius/key }
        volumeMounts:
        - { name: signing-key, mountPath: /run/asclepius, readOnly: true }
        readinessProbe:
          grpc: { port: 9090 }
          initialDelaySeconds: 5
        resources:
          requests: { cpu: '200m', memory: '256Mi' }
          limits:   { cpu: '1',    memory: '1Gi'   }
      volumes:
      - name: signing-key
        secret:
          secretName: asclepius-signing-key   # mounts as 0400 by default
Sidecar pattern. One Deployment per tenant boundary, not per app. All wrapped models in the same tenant share one ledger and one signing key. Cross-tenant mixing is a procurement red flag — the bundle for tenant A should never include rows from tenant B.
