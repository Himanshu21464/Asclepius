§01 · pre-flight · runbook · rollback · verify

Upgrading without breaking the chain.

The substrate is append-only by design — which makes upgrading fundamentally different from upgrading a typical service. Old entries must remain verifiable forever. Every migration on this page is structured to honour that invariant: pre-flight, runbook, rollback, verification.

M-001 shipped · v0.1.0 2026-04-19

v0.0.x → v0.1 · initial GA.

The first stable upgrade. The pre-GA on-disk format is replaced with the canonical-JSON spec; the genesis row is re-signed with the GA key; the keystore moves from on-disk to optional HSM-backed. Old chains are re-anchored, not re-written — the original entries remain bit-identical.

pre-flight

  • verify current chain · asclepius ledger verify
  • snapshot SQLite WAL to cold storage
  • publish current Merkle root in changelog feed

migrate

  • asclepius migrate --to 0.1
  • tool re-anchors v0.0.x rows under the v0.1 schema header
  • writes a schema.migrated entry with old + new roots

verify

  • verify forward · verify --range 0:tail
  • verify backwards from migration entry to genesis
  • publish new pubkey block in humans.txt

rollback

  • old binary still reads pre-migration rows (read-only)
  • restore SQLite snapshot · drop the post-migration entries
  • document the rollback as a schema.rollback event
# pre-flight · verify · snapshot
asclepius ledger verify --strict
asclepius ledger snapshot --out /var/asc/pre-0.1.db

# migrate
asclepius migrate --to 0.1 --confirm

# post-flight verify forward + back
asclepius ledger verify --range 0:tail

Backwards compatibility. The v0.1 binary can read v0.0.x WAL files end-to-end without migration if the operator chooses to remain on pre-GA semantics — the only forfeit is HSM support and the new schema's policies[] field. Migration is recommended but not forced.

M-002 planned · 2026 Q3 target v0.2.0

v0.1 → v0.2 · Postgres backend.

SQLite is the right default for single-tenant deployments. Multi-tenant operators want a real database. v0.2 introduces a pluggable backend interface; the SQLite implementation remains; a Postgres implementation ships alongside. The on-disk format does not change — only the storage engine.

pre-flight

  • provision Postgres 15+ · enable WAL archiving
  • verify SQLite chain · snapshot to cold storage
  • capacity plan: 2× expected entry rate for migration window

copy-forward

  • asclepius backend export --from sqlite --to pg-stream
  • reads SQLite tail-forward, streams to Postgres in entry order
  • preserves seq, prev_hash, sig bit-identical

cutover

  • flip read/write target · backend.primary=pg
  • SQLite remains read-only mirror for 30 days
  • verify both backends agree on tail Merkle root

rollback

  • flip backend.primary=sqlite
  • SQLite mirror has every row up to cutover; no data loss
  • any post-cutover entries on PG can be re-streamed back

Single-writer invariant. Even with Postgres as the backend, the runtime preserves single-writer semantics per ledger. Postgres unlocks multi-tenant parallel ledgers, not multi-writer per ledger. If you genuinely need multi-writer-per-ledger, talk to us — that is its own ADR.

M-003 planned · 2026 Q4 target v0.3.0

v0.2 → v0.3 · gRPC sidecar GA.

The asclepius-svc sidecar exits beta. The wire format (proto/asclepius/v1/sidecar.proto) is frozen at v1; future additions must be backward-compatible. The sidecar's protobuf contract is now an additional public artefact alongside the C++ API.

pre-flight

  • review proto v1 surface for client-side compatibility
  • confirm TLS termination configured
  • plan rate limits per tenant

deploy

  • deploy sidecar alongside runtime · share ledger
  • direct net-new traffic to sidecar; native API stays available

verify

  • traffic via sidecar produces entries identical to native
  • compare exported bundle hashes for parity

rollback

  • sidecar is purely additive · stop traffic, redirect to native
  • no on-disk effects to undo
M-004 research · 2027 target v1.0.0

v0.3 → v1.0 · ML-DSA dual-sign window.

The post-quantum migration. ADR-010 (proposed) outlines the destination: every entry signed with both Ed25519 and ML-DSA-65 during a multi-year transition window, with verifiers accepting either. Old entries keep their original signatures; new entries gain a second one.

pre-flight

  • generate ML-DSA-65 key in HSM (per ADR-010)
  • countersign new pubkey with current Ed25519 key
  • publish to humans.txt + RFC 9116 well-known

activate dual-sign

  • keys add --scheme ml-dsa-65 --key-id PQ-1
  • future entries gain a second sig.ml-dsa-65: field
  • old entries are not touched · old sigs remain verifiable forever

extend verifier

  • publish updated verifier binary supporting both schemes
  • auditor tooling tested against synthetic dual-signed corpus
  • document acceptance: "either signature suffices to verify"

retire (much later)

  • once ecosystem PQ-ready, deprecate Ed25519 emission
  • old entries keep their Ed25519 sigs; verifier still accepts them
  • this step is years out and will be its own ADR

Why the long window. Substrate-grade durability means a v0.1 bundle exported today should remain verifiable in 2046. Dual-signing gives the world time to get PQ tooling into HSMs, OS distributions, auditor binaries, and legal-discovery workflows — without anyone losing the ability to verify the past.

The chain survives the substrate.

Migrations exist to evolve the runtime without invalidating the record. Every upgrade above preserves the invariant that any entry committed by any prior version remains verifiable by any future version. If a proposed migration cannot meet that bar, it does not ship. That is the substrate's promise to the auditor.
