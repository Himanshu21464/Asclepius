Every choice, written down.

A substrate that asks operators to trust it should be able to explain why it picked the primitives it did. The records below are the "why" — the trade-offs we considered, the alternatives we rejected, and the consequences we accepted. If we ever change one, that becomes a new record, not an edit to an old one.

ADR-001 2026-01-12 accepted

Ed25519 over RSA / ECDSA.

context

The substrate signs every ledger entry, every evidence-bundle manifest, and every key-rotation record. The signature scheme has to be fast on modern CPUs (no HSM round-trip on the hot path), small on the wire (one sig per entry, often millions of entries), and well-understood by auditors. RSA, ECDSA-P256, and Ed25519 were the candidates.

decision

Use Ed25519 via libsodium crypto_sign_*. 64-byte detached signatures, deterministic, batch-verifiable. Public keys are 32 bytes; private keys 64 bytes (seed + pubkey).

consequences

  • ≈ 70 µs sign + 130 µs verify on commodity CPUs · zero round-trip
  • deterministic sigs → same input + key always produces same output (auditable)
  • no parameter choices to misuse (vs ECDSA's nonce reuse risks)
  • HSM support: most modern HSMs ship Ed25519 (AWS KMS Q3 2024)
  • no native FIPS-140-3 cert at v1.0; ML-DSA migration is queued (ADR-010)

alternatives considered

  • RSA-2048-PSS. 256-byte sigs blow up bundle size; 1.5 ms sign latency; legacy compatibility was not a real requirement.
  • ECDSA-P256. Acceptable, but nonce-reuse footguns and non-deterministic sigs make replay-comparison harder.
  • Schnorr / BIP-340. Excellent crypto, but ecosystem mostly biased to Bitcoin contexts; libsodium support is the operational tiebreaker.
ADR-002 2026-01-12 accepted

BLAKE2b-256 over SHA-256.

context

Every entry hashes its previous entry; every payload is hashed before signing; bundles include per-file hashes. The hash function runs orders of magnitude more often than the signature. We want the fastest sound 256-bit hash available to libsodium.

decision

Use BLAKE2b-256 via libsodium crypto_generichash. Produces 32-byte outputs, vectorisable, tree-hashable for bundle manifests. Outputs are encoded with the prefix blake2b: on the wire so verifiers can negotiate hash agility later.

consequences

  • ≈ 1 GB/s/core hashing throughput · the ledger never stalls on hash
  • no length-extension issues (unlike vanilla SHA-256)
  • hash agility planned: the prefix lets us add SHA-3 / BLAKE3 later
  • not FIPS-140-3 certified — same caveat as Ed25519

alternatives considered

  • SHA-256. Universally available, FIPS-friendly, ~3× slower for our shapes; we leave it as a future negotiable.
  • SHA-3 / Keccak-256. Slower in software than BLAKE2b without hardware acceleration; not yet pervasive in HSMs.
  • BLAKE3. Faster, parallelisable, but younger; we prefer the Bernstein lineage's track record for v1.
ADR-003 2026-01-18 accepted

SQLite WAL as the default backend.

context

The ledger needs durable, append-mostly storage that runs in a sidecar, survives a hard reset, supports many readers / one writer, and ships with the binary. Auditors expect to be able to inspect the data without installing a database server.

decision

Use SQLite in WAL mode as the default backend. One entries table, one keys table, one policies table. The WAL gives us crash-safe writes and concurrent reads; .dump produces a textual export auditors already know how to read.

consequences

  • single-file ledger trivially replicates (cp works)
  • no separate process to run, monitor, or upgrade
  • single-writer is a real limit; ADR-Postgres (TBD) will lift it for multi-tenant
  • WAL must be checkpointed periodically · runbook PB-07 handles partition recovery

alternatives considered

  • Postgres. Right answer at scale, but mandatory dependency at v1 would gate on every customer's DBA team. Planned as a pluggable v0.2 backend.
  • LMDB. Faster reads, but auditor ergonomics are worse — no SELECT, no tooling.
  • Custom append-only file. We are not in the business of writing yet another WAL.
ADR-004 2026-01-22 accepted

C++20 over Rust for the core.

context

The substrate has to embed in EHRs, gRPC sidecars, Python wheels, and eventually Java/JVM hosts. It has to interop with libsodium, sqlite, protobuf, and mature C ABIs. It has to be fast and predictable. Rust and C++20 were the credible options.

decision

Use C++20 with a strict subset (no exceptions on the hot path, no RTTI in public headers, no global state). Treat it like "Rust without the borrow checker" — turn on every warning, run ASan + UBSan in CI, use Pimpl for ABI stability (ADR-007).

consequences

  • libsodium / SQLite integration is zero-friction
  • pybind11 → first-class Python wheels in 200 lines of binding code
  • JNI, swift-bridge, and gRPC all consume the C ABI directly
  • memory-safety burden falls on us · sanitisers are not optional
  • future Rust port is not foreclosed — the public C ABI is the contract

alternatives considered

  • Rust. Better safety story, but the ecosystem for libsodium / sqlite / pybind-equivalent (PyO3) was less mature at the time we shipped, and the C++ talent pool in clinical software is the operational reality.
  • Go. Excellent for sidecars, weak for hot-path latency variance and embedability.
ADR-005 2026-02-04 accepted

Canonical JSON over CBOR.

context

Entries are signed by hashing a serialised form. The serialised form has to be byte-identical across writers and verifiers, decades apart. JSON is human-readable; CBOR is compact and binary; either can be canonical with effort.

decision

Use canonical JSON: UTF-8, sorted keys, no insignificant whitespace, integer or scientific-notation numbers, escaped strings per RFC 8259. Implementation: a single asc::canon::encode() function with property-based tests.

consequences

  • auditors can jq a bundle straight off disk
  • no separate decoder for human inspection vs verification
  • ~1.5× the size of equivalent CBOR · acceptable for the volume
  • numeric precision corner cases are explicit · canonical form pins them

alternatives considered

  • CBOR. Smaller, faster to parse, but tooling gap for auditors. Reserved as a potential v0.3 wire-only encoding.
  • Protobuf. Tied to a schema registry; bad for forward-compat over decades.
  • MsgPack. Same wire-vs-tooling argument as CBOR, less standardised.
ADR-006 2026-02-15 accepted

USTAR tar for evidence bundles.

context

Bundles are the artefact handed to regulators, peer reviewers, and auditors. They have to open with software that already exists on every operating system, decades from now. They must include an integrity manifest so a recipient can detect tampering offline.

decision

Use USTAR tar with a top-level manifest.json containing BLAKE2b-256 of every other file plus an Ed25519 signature. Recipients verify with stock tar, jq, and openssl — or with the published verify_bundle binary.

consequences

  • opens on macOS, Linux, BSD, Windows (built-in) without our tooling
  • uncompressed by default · bundles are deterministic byte-for-byte
  • USTAR's 100-char filename limit is a hard constraint; we enforce it
  • bundle size is real (50–200 MB typical) · acceptable for non-streaming use

alternatives considered

  • ZIP. Equally portable, but its manifest semantics are weaker; no canonical ordering guarantees.
  • OCI image format. Designed for container layers, not legal-grade artefacts; tooling assumes container runtime.
  • Custom binary format. Auditors should never have to install something to read evidence.
ADR-007 2026-02-22 accepted

Pimpl as the ABI strategy.

context

Asclepius binaries embed in long-lived clinical software. Rebuilding every consumer when our internals change is not acceptable. We need a stable C++ ABI for the public types — Runtime, Ledger, KeyStore, Inference — across at least minor versions.

decision

Use the Pimpl idiom for every public class: the header declares only forward-declared class Impl; and a std::unique_ptr<Impl>. Implementations are in the .cpp; consumers see only opaque pointers. Method additions require a recompile of consumers; layout changes do not.

consequences

  • internal refactors do not break consumer ABI
  • one heap allocation per public object (acceptable; objects are long-lived)
  • destructors must be defined in the .cpp where Impl is complete
  • inline-only optimisations are forfeit at the public boundary

alternatives considered

  • Header-only. Fastest, but every internal change is a downstream rebuild.
  • C ABI shim. Stronger guarantee, but loses RAII and ergonomic types at the boundary; reserved for Java/Swift bindings.
ADR-008 2026-03-01 accepted

Result<T,Error> over exceptions.

context

The runtime sits on the latency-sensitive path of every clinical AI call. Exceptions in C++ have non-trivial, hard-to-predict cost when thrown, especially across DLL boundaries, and they make control flow opaque to reviewers. Auditability requires that every fallible call be visibly handled.

decision

Use a Result<T, Error> type for every fallible public API. Internal code may throw, but all throws are caught at the public boundary and converted. Errors carry a code, a message, and a cause chain.

consequences

  • every error path is an explicit branch — easy to audit / fuzz
  • no stack-unwinding cost on the hot path
  • callers must check .ok() · linter enforces unused-result
  • cross-language bindings translate cleanly (Python raise, gRPC status)

alternatives considered

  • Exceptions everywhere. Fast in the no-throw case, but observability and FFI hostile.
  • Error codes only. Loses the "carries a value or carries an error" duality that Result makes natural.
  • std::expected. Same shape; we may switch to it once C++23 is the floor.
ADR-009 2026-03-08 accepted

Strong typedefs for IDs.

context

The substrate juggles many id types: ActorId, PatientId, EncounterId, ModelId, TenantId. They are all effectively strings, but mixing them is a category of bug that is catastrophic when it surfaces in a clinical context. One swapped argument should not be a quiet compile.

decision

Each id is a distinct type — a struct wrapping a std::string, with explicit construction and no implicit conversion to or from string. Comparison operators are defined within the type. Bindings translate to NewType-style classes (Python, mypy-friendly).

consequences

  • argument swaps fail to compile, not at 3 a.m.
  • some boilerplate at construction sites · acceptable
  • serialisation goes through explicit codec functions, not implicit casts

alternatives considered

  • Bare std::string. Cheaper, but the bug class is unacceptable for the deployment target.
  • Phantom types. Considered; the overhead is similar but the diagnostics are worse.
ADR-010 2026-04-19 proposed

ML-DSA migration path (post-quantum).

context

Ledger entries are durable evidence that a peer-reviewer might still verify in 2046. Cryptographically relevant quantum computers may exist by then. Ed25519 falls to Shor's algorithm. The substrate needs a forward-compatible posture.

decision (proposed)

Plan a v1.0 migration to ML-DSA (formerly Dilithium, FIPS 204). The wire format already prefixes signatures with a scheme identifier (ed25519:), so adding ml-dsa-65: is a compatibility-preserving extension. Bundles will dual-sign for a transition window; verifiers accept either.

consequences

  • bundle size grows substantially (~3 KB per ML-DSA-65 sig)
  • HSM support landscape is uneven · monitor through 2026
  • BLAKE2b is already PQ-safe (collision resistance only) — no hash migration
  • verifier tooling adds ~200 KB for the ML-DSA implementation

alternatives considered

  • SLH-DSA (SPHINCS+, FIPS 205). Hash-based, conservative, but signatures are 8–17 KB; bundle size becomes painful.
  • Falcon / FN-DSA (FIPS 206 draft). Smaller sigs, but floating-point implementation hazard.
  • Wait. Considered and rejected — substrate-grade durability requires we are ahead of the migration, not behind it.

What an ADR is, and is not.

An ADR is a record of a decision at a point in time. It is not a promise that the decision is forever right. When a decision changes, the old ADR is marked superseded and a new one is written — decision history is, like the ledger itself, append-only.

Writing them down is the cheapest cost. Not writing them down is how institutional memory rots, and how a substrate becomes "load-bearing magic only the original maintainers can reason about."
