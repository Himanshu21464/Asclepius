Operators, at the desk.

For the people whose pager actually goes off. Day one through disaster recovery, in the order you'll meet them. No marketing words; the substrate is small enough that the runbook fits on one page.

If you're standing up a fresh Asclepius node, do §01. If you're already running and the dashboard is green, do §02 on a cron. If something paged, jump to §03. If the planet wants to know the math behind your capacity numbers, that's §04.

§01 day one

Day one.

First-deploy checklist. Walk it once per box. Two files matter: the SQLite ledger and the Ed25519 keypair sitting next to it. Everything else is recoverable from those two.

The two files.

asclepius.db
The SQLite-WAL ledger. Holds every appended row, every signature, the key registry, drift baselines, consent grants and revocations. Read-mostly in steady state; one writer per process. Mode 0640; owner is the runtime's service account.
asclepius.db.key
The Ed25519 keypair the runtime signs entries with. Mode 0600. Generated on first Runtime::open if absent. This file is the signing capability. Treat it like a TLS private key: backed up to an HSM-equivalent store, never checked into source, never world-readable.
asclepius.db-wal
SQLite's write-ahead log. Transient; rolls into asclepius.db on checkpoint. Don't back this up in isolation — the ledger and the WAL are one logical artifact.
asclepius.db-shm
SQLite shared-memory file. Same lifetime as the WAL. Excluded from backups.

Filesystem layout.

Pick a single directory and stay there. The default the docs assume: 

# recommended layout for a single-node sidecar
/var/lib/asclepius/
├── asclepius.db          # the chain
├── asclepius.db.key      # 0600, owned by service account
├── asclepius.db-wal      # transient
└── asclepius.db-shm      # transient

/etc/asclepius/
├── asclepius.toml        # runtime config (policies, drift baselines)
└── policies/             # DSL or YAML policy chain (when present)

/var/log/asclepius/
└── runtime.log           # stdout/stderr; structured JSON lines

Permissions checklist.

  1. Service account owns /var/lib/asclepius/; nobody else has write.
  2. asclepius.db.key is 0600 and owned by the service account; not world-readable, not group-readable.
  3. asclepius.db is 0640; the operator group can read for backups; nobody else.
  4. The directory is on a filesystem with fsync honoured. No network mounts; no shared volumes between nodes.
  5. SELinux / AppArmor profile (when present) labels the directory asclepius_data_t; the runtime binary is the only writer.

Backups on day one.

Before the first inference lands, set the backup. The chain verifies offline; backups verify offline. If the production ledger goes away tomorrow, the backup is your full history.

  • Cadence: hourly snapshot of asclepius.db (full file copy after a SQLite PRAGMA wal_checkpoint(TRUNCATE)). Daily snapshot kept for 90 days. Monthly for the regulator-retention horizon (often 6+ years).
  • Mechanism: sqlite3 asclepius.db ".backup /backup/asclepius-$(date -u +%Y%m%dT%H%MZ).db". Streams a consistent copy without locking writers. Do not cp a hot WAL.
  • Off-host: ship snapshots to encrypted object storage, separate trust domain. The snapshot is content-addressed by its head hash; the object key can be the head hex.
  • Key file: back up asclepius.db.key separately, into the HSM / KMS the rest of your private keys live in. Never co-located with the chain backup. (See §05 for why.)

verify the backup, not the chain Restore the latest snapshot to a scratch directory and run asclepius ledger verify against it. Do this on first deploy and quarterly thereafter. Backups that have never been restored are not backups; they're claims.

Who has read access.

The substrate's job is to make the chain's bytes portable. The chain is not secret — anyone with the public key can verify it; that's the design. So read access is a question about the operational artifact (the .db), not the information (the chain).

  • Service account: reads + writes; the runtime process and nothing else.
  • Operator group: reads, for backups and emergency triage. Two named humans, rotated quarterly.
  • Auditor / regulator: reads, on request, against a snapshot — never the live file. Hand them an evidence bundle (see §03).
  • Everyone else: nothing. The .key is only the service account.
§02 daily probes

Daily probes.

Three commands cover the steady-state. Run the first on every scrape; the second on a slow cron; the third on every deployment.

runtime health — every minute.

Cheap composite: ledger length, head prefix, key id, drift feature count, active inference count. The line a status page renders. 

$ ./build/apps/cli/asclepius runtime health /var/lib/asclepius/asclepius.db
  {
    "version":           "0.4.3",
    "ledger_length":     142091,
    "ledger_head":       "8e510bf4...c20e",
    "ledger_size_bytes": 1283604480,
    "key_id":            "a1f274e6",
    "drift_features":    8,
    "active_inferences": 3,
    "is_chain_empty":    false
  }

Wire runtime health behind your existing /healthz. Fail the probe on any of: process exit non-zero, active_inferences wedged at the same positive value across three consecutive scrapes, or ledger_size_bytes shrinking.

ledger verify — every hour.

Walks the chain, recomputes each entry's hash, re-checks each Ed25519 signature, confirms every prev_hash. The thing the regulator runs. A green here is the strongest single guarantee the substrate produces. 

$ ./build/apps/cli/asclepius ledger verify /var/lib/asclepius/asclepius.db
  OK · 142091 entries · head=8e510bf4...c20e · key=a1f274e6 · 14.2s

On a chain past a few million entries, switch to asclepius ledger verify --parallel for the background job and to runtime audit-spot-check --lookback 1024 for the per-minute startup gate. The spot-check verifies only the last 1024 rows; cheap, but enough to catch a corrupted tail before traffic resumes.

runtime self-test — every deploy.

Smoke test: hash a known input, sign and verify, append to an in-memory chain, verify again. Returns ok only if every component is wired correctly. Use as a deployment gate before traffic-shifting. 

$ ./build/apps/cli/asclepius runtime self-test
  ok · libsodium present · keystore round-trip · in-memory chain verifies

Cron sketch.

# /etc/cron.d/asclepius — runs as the service account
# every minute: scrape /healthz (your existing probe wraps `runtime health`)
# every hour: full chain verify, alert on non-zero
17 * * * *  asclepius  /usr/local/bin/asclepius ledger verify /var/lib/asclepius/asclepius.db || \
                       /usr/local/bin/page-oncall "ledger.verify failed on $(hostname)"

# every 6h: WAL checkpoint + content-addressed snapshot
3  */6 * * * asclepius  sqlite3 /var/lib/asclepius/asclepius.db \
                          "PRAGMA wal_checkpoint(TRUNCATE);" && \
                        sqlite3 /var/lib/asclepius/asclepius.db \
                          ".backup /backup/asclepius-$(date -u +\%Y\%m\%dT\%HZ).db"

# daily 03:14: verify the latest backup against itself
14 3  * * * asclepius  /usr/local/bin/asclepius ledger verify \
                          $(ls -t /backup/asclepius-*.db | head -1)

read the bytes once The substrate is designed so that an operator can produce every claim it ever made by reading the chain. If your dashboard and your ledger verify disagree, the chain is the ground truth — fix the dashboard, not the chain.

§03 when something paged

When something paged.

Incident response. Five subcommands cover ninety percent of what you'll do in the first twenty minutes after the alert fires.

Find the row.

Almost every incident starts with an inference id. The clinician complained about a draft; the EHR logged a refusal; a regulator wants the row for patient X on date Y. Pull the entry. 

# O(log n) lookup — pulls the inference.committed row
$ ./build/apps/cli/asclepius ledger find /var/lib/asclepius/asclepius.db \
        --inference-id 01HXVBS3DHTNG7KQNJZ4RFG7XK
  seq=128431 ts=2026-04-26T14:22:11.503Z actor=clinician:dr.demo
    model=ambient-scribe-v3 input=8e510bf4... output=fa14b6c7...
    policies=[phi_scrubber:allow, length_limit:allow]
    consent_token=ct_3f8a... purposes=[ambient_documentation]

# existence-only — cheap, doesn't materialize the row
$ ./build/apps/cli/asclepius ledger has-inference-id /var/lib/asclepius/asclepius.db \
        --id 01HXVBS3DHTNG7KQNJZ4RFG7XK
  true

What did this actor do today?

A clinician rage-pages the help desk; the on-call needs every inference that account authored in the past hour. tail-actor pulls them in one query, descending. 

$ ./build/apps/cli/asclepius ledger tail-actor /var/lib/asclepius/asclepius.db \
        --actor "clinician:dr.demo" --n 50
  142091  2026-04-26T14:22:11Z  inference.committed  ambient-scribe-v3
  142087  2026-04-26T14:18:43Z  inference.committed  ambient-scribe-v3
  142080  2026-04-26T14:11:02Z  inference.aborted    ambient-scribe-v3
  ...

Time + event-type, in one query.

Round-80 ergonomic: events_between takes a time window and an event type, and lands the composite filter in one indexed read. 

# every consent.revoked landed in the last 24h
$ ./build/apps/cli/asclepius ledger events-between /var/lib/asclepius/asclepius.db \
        --from "2026-04-25T14:00:00Z" --to "2026-04-26T14:00:00Z" \
        --event-type "consent.revoked"
  142091  2026-04-26T14:22:11Z  consent.revoked  patient=p_9f1a  token=ct_3f8a...
  141823  2026-04-26T11:08:22Z  consent.revoked  patient=p_a801  token=ct_b2cc...
  ...

For more general ad-hoc work, ledger filter accepts a predicate. It's slow (walks the chain); the indexed accessors above should always be preferred when they fit.

Hand it to a third party.

When the regulator (or your security team) asks for "every event for patient X during March," produce an evidence bundle. The bundle is a USTAR tar with the ledger excerpt, manifest, signed head, every inclusion proof, and the verifier instructions baked in. Verifiable offline. 

$ ./build/apps/cli/asclepius evidence bundle /var/lib/asclepius/asclepius.db \
        --patient "p_9f1a" --from "2026-03-01" --to "2026-04-01" \
        --out /tmp/p_9f1a-march.tar
  wrote /tmp/p_9f1a-march.tar · 814 entries · 4.2 MB · head=8e51...c20e

# the regulator runs this on a clean machine, no Asclepius needed
$ ./build/apps/cli/asclepius evidence verify /tmp/p_9f1a-march.tar
  OK · 814 entries · manifest signed by key=a1f274e6 · all hashes match

compact attestations asclepius runtime head-attestation emits the one-line head_attestation_json() — length, head, key id, fingerprint, signature on a single grep-friendly line. Pin it into a public timestamping service or your status page; if the chain is later challenged, the timestamped attestation bounds the worst-case backdating window.

If ledger verify failed.

This is the page that should never fire. If it does:

  1. Stop traffic to the runtime. A failed verify means the chain on disk no longer signs to the head it published. Until you know which row, do not append more.
  2. Find the break. asclepius ledger verify --bisect walks the chain in halves; reports the first seq whose hash or signature does not match. The break is at exactly that row.
  3. Compare against backup. Restore the previous-hour snapshot to scratch and run verify there. If the backup verifies and the live file doesn't, the integrity loss is bounded by the backup horizon.
  4. Capture an evidence bundle of the live file before any recovery action — the corruption itself is auditable. Then restore from the last verifying snapshot.
  5. Treat the key as suspect. If the break is at multiple discontiguous seqs, assume the key file was used outside the runtime. Rotate. (See §05.)
§04 capacity planning

Capacity planning.

Three numbers carry the math. Per-entry size, drift state, checkpoint cadence. With those three you can size a year of disk in your head.

Ledger growth, in bytes per inference.

A canonical inference.committed entry is roughly ~8 KB on disk after SQLite overhead — header, canonical body, prev_hash, signature, indexed columns. consent.granted / consent.revoked rows are smaller (~1.2 KB). Round to 10 KB per logical event for a worst-case sizing exercise.

For a department running 5,000 inferences a day: 

# back-of-envelope
5,000 inferences/day  ×  10 KB/event  =  50 MB/day
                                       1.5 GB/month
                                        18 GB/year

# with 6-year regulator retention
                                       108 GB / chain

The chain compresses well (Merkle prev_hashes are high-entropy but the canonical-body fields are not); a VACUUM after a quarter typically reclaims 5–10%. The on-disk total is linear in entries — no compaction magic, no archival tier. Plan for it as a flat-line.

Drift state.

DriftMonitor holds, per registered feature, two histograms (reference + current) and a small alert latch. Histograms are typically 64 or 128 bins of uint64_t. Round to 2 KB per feature in memory, persisted into the ledger only on baseline-reset.

  • 10 features → 20 KB resident, plus the per-inference observation cost (a single uint64_t increment).
  • 100 features → 200 KB. Still negligible.
  • If you find yourself registering thousands of features, you're using DriftMonitor wrong; switch to a metric per shard with embedding-drift on top (planned, post-1.0).

When to checkpoint.

SQLite's WAL mode rolls writes into a separate file and checkpoints back into the main file periodically. The runtime triggers an automatic checkpoint at 1,000 entries or 16 MB of WAL, whichever comes first. For a steady-state department, that's roughly one checkpoint per twenty minutes.

You should manually PRAGMA wal_checkpoint(TRUNCATE) before every backup, before any VACUUM, and immediately before a planned process restart. Backing up a hot WAL is the most common way to make a backup that doesn't verify.

watch for If asclepius.db-wal grows past ~64 MB and stays there, the auto-checkpoint isn't firing — usually because a long-running read transaction is blocking truncation. Find the reader (SELECT * FROM sqlite_master at the bottom of an analytics job is the usual culprit) and fix it; do not just delete the WAL file.

§05 disaster recovery

Disaster recovery.

Two questions. What can be recovered from the chain alone, and what can't. Knowing the difference sizes your backup strategy.

What the chain alone recovers.

  • Every inference. The full record of who asked what, when, what the model returned, which policies ran, with what verdicts, against which inputs. Reproducibility witness lives in the entry; nothing is lost.
  • Consent state. The runtime replays consent grants and revocations from the chain on startup; the in-memory ConsentRegistry is rebuilt from the audit log. The audit chain is the source of truth for consent.
  • Drift baselines. Persisted into the chain on every register_feature. Restart-recoverable.
  • Policy chain configuration. policy.config_change rows record every push / remove / rename. Replayable.
  • Aggregates. Stats, tenant counts, actor counts — all derivable from the chain in one pass; no separate state to lose.

What the chain alone cannot recover.

  • The signing key. If asclepius.db.key is gone and the public key is not registered elsewhere, you have a chain you can read but cannot extend with the same authoring identity. Append a new key.rotated entry signed by a fresh keypair; the regulator-facing trust now requires the old public key (cached in archived head_attestation_json outputs and in any evidence bundle ever emitted).
  • Off-chain bodies. If you opt into the encrypted body-store (post-1.0), the body is encrypted-at-rest in object storage. Lose that bucket and you lose the bodies, but the hashes remain — you can prove a body of given content existed, you just can't show it.
  • External timestamps. Rebuilding the chain from a backup re-creates the exact same row-hashes — but a third-party timestamp service that anchored a head at time T cannot re-issue the anchoring. Pin head_attestation_json outputs into your archive separately.

Key rotation.

Rotation is a planned event, not a recovery one. The procedure:

  1. Generate a new keypair on the same node (or in your HSM): asclepius keystore generate --out /var/lib/asclepius/asclepius.db.key.new.
  2. Append a key.rotated entry signed by the old key, naming the new public key. This is the cryptographic handoff. Regulators tracking your chain will see the rotation row and accept signatures from the new key for entries past that seq.
  3. Atomically swap: mv asclepius.db.key.new asclepius.db.key after a clean SIGTERM, restart the runtime.
  4. Archive the old .key file in your HSM under a "retired" tag. Don't delete it; you may need it to verify pre-rotation entries on a recovered backup.
  5. Re-emit a fresh attestation: asclepius runtime head-attestation. The new attestation references the new key id; pin it into your timestamping service.

Backup cadence, with the math.

  • Hourly snapshot of the live ledger. Worst-case data-loss horizon: 1 hour.
  • Daily snapshot kept 90 days, on independent storage. Worst-case integrity-loss horizon: 1 day.
  • Monthly snapshot kept for the regulator-retention horizon (often 6+ years), on cold storage with cryptographic head pinning.
  • Restore drill quarterly: pull the most recent monthly snapshot, restore to scratch, run asclepius ledger verify, write the timestamp into your DR registry. An untested backup is a wish.
§06 multi-tenant ops

Multi-tenant ops.

The substrate is single-node, but a single node can serve many tenants. Every entry carries a TenantId; every per-tenant query reads only that tenant's slice. The CLI commands below all accept --tenant; without it, you get the empty-tenant default scope, which is its own thing.

Per-tenant stats.

$ ./build/apps/cli/asclepius ledger stats-tenant /var/lib/asclepius/asclepius.db \
        --tenant "stmary-cardiology"
  entries          18421
  bytes_on_disk    192987136
  actors           14
  oldest           2026-01-04T08:11:02Z (seq 1024)
  newest           2026-04-26T14:22:11Z (seq 142091)

For per-customer billing, byte_size_per_tenant emits the per-tenant byte slice in one pass — useful when the commercial offering bills on retention.

Per-tenant queries.

  • asclepius ledger oldest-n-for-tenant --tenant T --n 10 — head-end mirror of tail_for_patient; useful for "how did this tenant onboard?"
  • asclepius ledger range-by-actor --tenant T --actor A — same shape, indexed.
  • asclepius consent stats-for-patient --tenant T --patient P — per-patient rollup, scoped.

the empty-tenant gotcha If a caller never set InferenceSpec::tenant, the entry lands under the default TenantId{""} — the empty tenant. ledger stats with no --tenant flag returns the global cross-tenant view, not the empty-tenant view. To inspect just the empty-tenant slice, pass --tenant "" explicitly. This bites teams during the first week of a multi-tenant rollout: every entry shows up under the empty default until the integration sets the tenant. Audit the empty-tenant slice on day one of the rollout; if it's nonzero past the integration, fix the caller, not the chain.

Tenant isolation.

Asclepius does not enforce tenant secrecy — anyone with read access to the .db can see every tenant's slice. Isolation lives one layer up: a sidecar per tenant, a separate .db per tenant, or application-layer authn before the runtime is reached. The substrate is the audit layer; the access-control layer is your job.

§07 what you'll page on

What you'll page on.

Four signals carry the alert traffic. Listed in descending severity. Each row: what the runtime emits, what it means, what to do.

  • P1 · drift.crossed

    One or more registered drift features moved into the severe band — DriftMonitor::any_severe() is true. Population the model is now serving has shifted enough that the held-out baseline no longer applies. Response: page the model owner, freeze the policy chain on the affected scope, run most_drifted_feature for the drill-down, clear_alerts() only after acknowledgement.

  • P1 · ledger.verify failure

    ledger verify non-zero exit. The chain on disk no longer signs. Treat as a candidate integrity incident. Response: stop traffic, bisect to the break, capture an evidence bundle of the corrupted live file, restore from last verifying snapshot, consider the key suspect if the break pattern is non-contiguous. Procedure in §03.

  • P2 · consent_missing

    begin_inference rejected with denied on a consent.permits check. The clinician has tried to invoke a tool the patient has not consented to — sometimes legitimate (consent expired); sometimes a misconfiguration of the EHR-side consent flow. Response: recently_revoked(window) tells you whether the patient's withdrawal landed; if not, escalate to the consent-flow owner.

  • P2 · slow inference

    elapsed_ms on an inference.committed entry crossed the SLO. Most often the model behind the wrap, not the substrate — Asclepius adds <200 µs in the hot path. Response: walk tail_by_actor on the affected service for the slow window; pair with elapsed_ms quantiles from the metric registry.

  • P3 · key.rotated

    Informational. A key.rotated entry was appended; the runtime is now signing with a new key. Confirm the rotation was planned; if not, treat as a P1 integrity event and assume the old key is compromised.

  • P3 · WAL stuck

    asclepius.db-wal exceeds 64 MB and is not draining. A long-running read transaction is blocking checkpoint truncation. Find the reader; do not delete the WAL.
