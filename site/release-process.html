§01 · cut · sign · SBOM · advise

How a release actually happens.

Substrate-grade software ships under a substrate-grade process. Every Asclepius release follows the same six-stage pipeline — version cut, build, sign, attest, publish, advise — and every step is reproducible from the git tag. If a step doesn't pass, the release doesn't ship. No hot-fix lane skips signing.

§ 01 · cut

Cut a release branch.

A release starts with a release-engineer cutting release/x.y from main at a specific commit. The same engineer is then barred from approving the release — single-engineer release is forbidden by policy.

tag candidate

  • annotated tag v{major}.{minor}.{patch}-rc.{n}
  • tag is signed with the release-engineer's PGP key
  • release branch protected — no force-push, no direct merge

freeze the changelog

  • Keep-a-Changelog format · added / changed / removed / fixed / security
  • each line links to a commit, an issue, or a PR
  • RSS / JSON / Atom feeds regenerate from changelog.html

SBOM

  • generate CycloneDX-1.6 SBOM via sbom-tool
  • covers libsodium, sqlite, abseil, protobuf, doctest, pybind11
  • checked into artifacts/sbom-{ver}.json

migration manifest

  • cross-reference any ADR or migration record (migrations)
  • verify migration tool runs on a v{prev} fixture corpus
  • attach migration result hashes to the release
# cut a release candidate
git switch -c release/0.2
git tag --sign v0.2.0-rc.1 -m "v0.2 RC1: pluggable backend"
asclepius release sbom --out artifacts/sbom-0.2.0.json
asclepius release manifest --since v0.1.0
§ 02 · build

Build, reproducibly.

Two independent build runners fetch the same tag, build with the same toolchain pin, and emit byte-identical binaries. Any divergence halts the release. The toolchain itself is pinned by hash in cmake/toolchain.cmake.

platformcompilerlibsodiumoutput
Linux x86-64 GCC 13.3 · libstdc++1.0.20 (pinned)asclepius-linux-x86_64.tar.zst
Linux aarch64 GCC 13.3 · libstdc++1.0.20 (pinned)asclepius-linux-aarch64.tar.zst
macOS arm64 Apple clang 16 1.0.20 (pinned)asclepius-macos-arm64.tar.zst
Python wheels cp 3.10–3.13 · manylinux_2_28vendoredasclepius-{ver}-{cp}-{abi}-{plat}.whl
OCI image distroless · multi-arch vendoredghcr.io/Himanshu21464/Asclepius:{ver}

What "reproducible" means here. Two runners on different machines in different regions, same commit, same toolchain, must produce bit-identical output. We use SOURCE_DATE_EPOCH, fixed library order, and a sorted file system listing. Byte equality or it doesn't ship.

§ 03 · sign

Sign with two mechanisms.

Every released artefact is signed twice: a long-lived PGP key for human verification, and a Sigstore (cosign) keyless signature for ecosystem interop. Recipients can verify either; we recommend both.

mechanismkeyverifier
PGP detachedfingerprint A1F2 9C4E B8D1 74E6 · 2A55 0CCB 7F12 BB37 5E4A C9D0gpg --verify
Sigstore OIDC-bound · GitHub Actions identity cosign verify-blob
SLSA provenanceauto-generated by reproducible-build runner slsa-verifier
 
# verify a release tarball end-to-end
gpg --verify asclepius-0.2.0-linux-x86_64.tar.zst.asc \
             asclepius-0.2.0-linux-x86_64.tar.zst

cosign verify-blob \
       --certificate-identity-regexp "https://github\\.com/asclepius-ai/.+" \
       --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
       --signature asclepius-0.2.0-linux-x86_64.tar.zst.sig \
                   asclepius-0.2.0-linux-x86_64.tar.zst

slsa-verifier verify-artifact \
       --provenance-path provenance.intoto.jsonl \
       --source-uri github.com/Himanshu21464/Asclepius \
       --source-tag v0.2.0 \
                   asclepius-0.2.0-linux-x86_64.tar.zst
§ 04 · attest

Attest the supply chain.

Releases include in-toto attestations covering the full build pipeline — source revision, build environment, dependencies, tests run, signing keys touched. Targeting SLSA Level 3 at v1.0; currently SLSA Level 2.

source

  • commit SHA · branch protection log
  • two-reviewer requirement enforced
  • signed-tag commit identity

build env

  • distroless build container · pinned by digest
  • compiler binaries pinned by SHA-256
  • recorded in attestation predicate

tests

  • 52 doctest C++ cases · 110 JS site tests · all green
  • conformance suite L1/L2/L3 attestation hash
  • bench numbers regenerated to assets/bench.json

deps

  • SBOM hash bound into provenance
  • vulnerability scan output (osv-scanner) attached
  • any High/Critical = release blocker
§ 05 · publish

Once signed and attested, the release publishes simultaneously to every consumer channel. Single-source-of-truth for the artefacts is the GitHub release; mirrors point at it.

§ 06 · advise

Advise the downstream.

Releases that fix a security issue ship with an advisory. Releases that introduce a deprecation ship with a migration window. Releases that change the audit-entry schema ship with a major-version bump and a dual-support window. Operators are never surprised.

advisory severity

severityfix indisclosure
critical14 days from triageoperator advisory + CVE within 7 days of patch
high 30 days advisory bundled with release
medium 60 days advisory in next quarterly roll-up
low next release changelog entry suffices

deprecation policy

kindwindowsignalling
public C++ API1 minor version of co-existence[[deprecated]] attribute · changelog · migration guide
CLI flag 1 minor version warning printed · removal one version later
audit-entry schema fieldmajor version bump schema versioned · old verifier still works
signing schememulti-year dual-sign window see ADR-010

Why we're strict about this. A substrate that surprises its consumers is not a substrate — it's a competing project that happens to ship in your binary. The release process exists so operators can rely on Asclepius for years without re-reading docs every quarter.

The pipeline is the contract.

Anyone can clone the repo, check out a tag, and rebuild the release from scratch — using the same toolchain, the same flags, the same hash of the same libsodium. Reproducibility is not a marketing property. It is the substrate's promise that the bytes you run are the bytes the maintainers signed, and nothing else.
