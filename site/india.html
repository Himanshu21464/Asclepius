The India profile.

Asclepius is not an India-only product. The kernel is the substrate underneath any clinical-AI tool, anywhere it runs. The India profile is one optional configuration of that substrate — three primitives and six Purpose values that an Indian deployment can switch on, while a US, EU, or anywhere-else deployment ignores them entirely.

We landed it in round 90 because India's national digital-health infrastructure is a real, working consent fabric — ABDM, the DPDP Act, a stable HIP/HIU vocabulary — and a kernel that wants to be the substrate underneath clinical AI cannot pretend that fabric does not exist. Document the seam, not the policy.

§01 the substrate

Substrate, not jurisdiction.

The kernel ships one runtime, one ledger, one consent registry. The India profile adds three new types — ConsentArtefact, FamilyGraph, EmergencyOverride — and six new Purpose enum values. Nothing else changes. A deployment that never instantiates these types is unaffected; a deployment that does instantiate them gets the ABDM-shaped wire format, multi-party consent, and DPDP-aligned break-glass without forking the runtime.

We do not bake an ABDM Gateway client, an HIP adapter, an HIU consumer, or an Aadhaar verification path into the kernel. Operators wire those above the substrate, the same way they wire any other vendor adapter. The kernel codifies the contract; the operator codifies the operator workflow.

analogy Linux ships iptables primitives without prescribing a corporate firewall policy. The India profile adds primitives that fit the shape of ABDM consent semantics; what an Indian deployment does with them stays the operator's problem.

§02 consent artefacts

ABDM consent artefacts.

ConsentArtefact is a JSON-serialisable shape that mirrors an ABDM consent-artefact record. It is the wire format a kernel deployment exchanges with HIPs (Health Information Providers), HIUs (Health Information Users), and the ABDM Gateway. It is intentionally not the same type as ConsentToken: the registry token is the kernel's authoritative in-memory record; the artefact is the externalised, signable, transmittable shape. Bidirectional mapping is provided so a token can travel out and a fetched artefact can travel in. 

// authoritative token in the registry → outbound ABDM artefact
asclepius::ConsentArtefact a = asclepius::artefact_from_token(
    token,                     // kernel-side ConsentToken
    "hiu:apollo-blr",           // requester id (HIU)
    "hip:fortis-tnk",           // fetcher id (HIP)
    "abdm:abc-123-def");         // ABDM artefact id

std::string wire = asclepius::to_abdm_json(a);
// {"artefact_id":"abdm:abc-123-def","patient":"...", "purposes":[...], "status":"granted", ...}

// inbound artefact from a peer → kernel-side token
auto roundtrip = asclepius::from_abdm_json(wire).value();
asclepius::ConsentToken t = asclepius::token_from_artefact(roundtrip);

Status priority on the artefact mirrors what the registry already encodes: granted, revoked, expired. A revoked-and-expired token serialises as revoked — the verb the patient invoked is the one the artefact carries.

§03 family graph

The family graph.

A consent registry that only knows (patient → grant) cannot answer the question Indian healthtech actually asks every day: can this adult consent on behalf of their elder parent? FamilyGraph records edges from a proxy to a subject, tagged with one of four relationships. A token issued by a proxy is honoured iff a valid edge exists in the graph; the runtime mirrors edge writes to the ledger so graph history is reconstructable from the chain.

  • FamilyRelation::adult_child_for_elder_parent — the canonical Indian eldercare path.
  • FamilyRelation::parent_for_minor — paediatric consent, parent on behalf of child.
  • FamilyRelation::legal_guardian_for_ward — court-appointed or institutional guardianship.
  • FamilyRelation::spouse_for_spouse — spousal consent in incapacity scenarios.
asclepius::FamilyGraph fg;
fg.record_relation(adult_child, elder_parent,
                   asclepius::FamilyRelation::adult_child_for_elder_parent);

if (fg.can_consent_for(adult_child, elder_parent)) {
    // proxy is authorised; honour the token they issued
}

The graph is policy-neutral about which relationships an operator chooses to recognise — an institution can choose to accept only parent_for_minor and reject the rest, or run with all four. The kernel just records and looks up edges.

§04 emergency override

Break-glass with a backfill window.

EmergencyOverride is the kernel primitive for DPDP-aligned deferred consent. In a clinically-justified emergency — a road-traffic admission, an unconscious patient, an ED triage decision — a clinician may access records before consent can be granted in the moment. The kernel records the access immediately and computes a backfill_deadline of activated_at + window (default 72 hours). Inside the window, an operator must file a backfill evidence id; outside the window, is_overdue() reports the token and dashboards surface it. 

asclepius::EmergencyOverride eo;
auto tok = eo.activate(actor, patient,
                      "RTA, GCS 7, ED triage").value();

// inside the 72h window, operator backs it up:
eo.backfill(tok.token_id, "signed-clinical-note:0x4a");

// or, if no one filed:
if (eo.is_overdue(tok.token_id)) {
    // surfaces in dashboards; overdue_backfills() returns the row
}

The kernel is intentionally policy-neutral about what satisfies the backfill. A signed clinical note, a scanned consent form, an inbound ABDM consent-artefact id — all are equally valid as far as the kernel is concerned. What counts as evidence is the operator's compliance question; that the backfill happened within the window is the kernel's question.

configurable, not hard-coded The 72-hour default exists because it is what DPDP § 7 reads as currently. set_backfill_window() moves it. If regulation tightens to 48 hours or loosens to 96, an operator flips one call and existing tokens keep the deadline they were born with.

§05 india purposes

Six new Purpose values.

The kernel's Purpose enum was eight values for the original kernel surface. Round 90 added six more, each chosen to match a workflow that India healthtech needs to express without stretching an existing purpose past its meaning. Adding a Purpose is a governance event (see ADR-014); the additions are backwards-compatible — clients that don't request the new purposes are unaffected.

prescription_resolution
Parse a prescription and resolve it to NPPA / PMBJP catalogues — the price-comparison and generic-substitution path.
second_opinion
Share a case dossier for a structured second opinion; consent is bounded to the dossier and the named consulting party.
specialist_referral
Share a case dossier with a tier-2/3 specialist for tele-consult; the referral path is distinct from a free-form second opinion.
billing_audit
Audit an itemised hospital bill against a published reference catalogue; consent here is for the bill, not the underlying clinical data.
longitudinal_outcomes_research
Per-patient longitudinal outcome tracking — a cohort ledger purpose distinct from one-off research grants.
emergency_clinical_access
DPDP § 7 break-glass: deferred-consent clinical access. Pairs with EmergencyOverride; this is the purpose recorded on the inference, the override is the permission covering it.
§06 dpdp alignment

DPDP § 7, codified.

The Digital Personal Data Protection Act, 2023, § 7, allows clinical processing of personal data without prior consent in circumstances where consent cannot be obtained — most centrally, a medical emergency where the patient cannot grant it in the moment. The clause is not a free pass: it expects a recorded justification and a path to retrospective notice or consent once the situation permits.

The kernel codifies the requirement, not the regulation. EmergencyOverride records the activation immediately, computes a deadline at activation time, and exposes is_overdue() and overdue_backfills() so dashboards and audit alerts cannot quietly miss a backfill. That is what the substrate guarantees.

The window is configurable — set_backfill_window() takes a std::chrono::seconds — so a deployment can match whatever DPDP rule is in force, or a state-level health rule, or an institutional policy more conservative than either. The kernel does not ship a position on what the right number is; the 72-hour default reads the regulation as it stands. Operators who disagree are one call away from their own number.

not legal advice The substrate produces the artifacts a DPDP regime expects; compliance is a human job. Counsel reads the bundle; they do not run the runtime.

Evidence bundles.

Rounds 91–96 layered five signed evidence shapes on top of the round-90 consent primitives. Each is a substrate object: the kernel produces it, signs it, verifies it. Operators decide what to attest; the kernel guarantees the attestation cannot be silently mutated.

  • HumanAttestation Signed clinician sign-off (round 92). actor + subject_kind + subject_id + statement, anchored in the ledger via events::human_attestation. Backs second-opinion review, triage override, billing approval — any workflow that requires a named human on a decision.
  • TeleConsultEnvelope Two-party signed encounter (round 95) for SpecHub. Patient and clinician each sign over the consult metadata, video hash, and transcript hash. is_fully_signed() is the integrity probe.
  • BillAuditBundle Auditor-signed bill audit (round 95) for BillBuster. Each BillLineFinding is classified consistent / over_billed / under_billed / unknown_item against a named reference table (CGHS-2025, PMJAY, operator-private). The kernel ships classify_line with the 1.5x / 0.5x band convention; operators that need different bands feed pre-classified findings.
  • SampleIntegrityBundle Chain-of-custody attestation (round 96) for VanRoute. sample_id + collector + checkpoint trail + result hash. cold_chain_intact() is true iff every checkpoint was within spec.
  • CarePathAttestation Operator-side signed proof (round 96) that an inference call was evaluated against an access::Constraint (Sakhi-style female-only, on-device-only, language allowlist, role gate) and either allowed or denied. Replays the kernel's care-path decision at audit time.

Operational knobs.

  • events:: well-known codes Round 92 codifies the canonical 12 event-type strings (rx.parsed, triage.decision, consult.tele.closed, …) so dashboards and conformance tests agree on what counts. Free-form event_type still works.
  • CalibrationMonitor Round 92 sensitivity / specificity / PPV / NPV tracking against a configured floor (with tolerance and min-samples). Used by TriageMate to surface "model is below the agreed sensitivity floor" before the regulator sees the audit.
  • CohortLedger Round 93 longitudinal observation chain. Per-patient or per-metric outcome tracking; stats_for_metric returns count/mean/stddev/min/max for any metric across the cohort.
  • access::Constraint Round 93 composable predicate (staff_gender / device_mode / language allowlist / role). Sakhi-style female-only, TriageMate-style on-device-only, language-restricted deployments, all the same primitive.
  • asclepius india CLI Round 94. india validate-bundle for ABDM Consent JSON; india verify-attestation for HumanAttestation JSON; india events to enumerate the canonical event codes. Exits non-zero on any verification failure — suitable for CI gates.
  • FHIR ABDM bridge Round 91. bundle_from_artefact / artefact_from_bundle round-trip a ConsentArtefact through a FHIR Bundle wire shape with well_formed_bundle as the precondition gate, and HIP / HIU / ABHA identifiers carried in meta.tag.

What we do not do.

  • not pricing The kernel does not bake any pricing schedule into billing_audit. Operators pull NPPA / PMBJP / state catalogues; the kernel records the comparison.
  • not URLs No ABDM Gateway URL, no HIP discovery endpoint, no Aadhaar verification path is hard-coded. Adapters go above the substrate.
  • not workflows Whether a backfill needs a signed note or a scanned form is the operator's compliance question. The kernel just enforces that some evidence id is filed before the deadline.
  • not jurisdiction An EU or US deployment can build, link, and run the kernel without ever touching the India profile. The new types compile in; instantiating them is opt-in.
