Quickstart, for builders.

From git clone to a verified, signed ledger entry on disk — in roughly five minutes on a laptop. No vendor account, no sign-up, no telemetry. A C++ toolchain and libsodium are the whole prerequisite.

This page is the on-ramp for first-time builders. If you wanted the plain-language version of what Asclepius is, that lives at /explained. If you want to skip the local build entirely and play in a browser tab, that's /demo. Otherwise, pick a terminal and read on.

§01 in one minute

In one minute.

Three commands. Paste them into a fresh terminal and read on while the build runs. 

# 1. clone
$ git clone https://github.com/Himanshu21464/Asclepius.git
$ cd Asclepius

# 2. build (RelWithDebInfo, ~90s on a 2024 laptop)
$ cmake -S . -B build -DCMAKE_BUILD_TYPE=RelWithDebInfo \
        -DCMAKE_POLICY_VERSION_MINIMUM=3.5
$ cmake --build build -j

# 3. run the wrap_scribe example
$ ./build/examples/01_wrap_scribe/example_wrap_scribe
  wrap_scribe · ok · seq=1 · head=8e510bf4...c20e

prereqs build-essential, cmake (≥ 3.20), ninja-build, libsodium-dev, libsqlite3-dev. On Debian/Ubuntu: sudo apt install build-essential cmake ninja-build libsodium-dev libsqlite3-dev. On macOS: brew install cmake ninja libsodium and SQLite via the system framework.

If the third line printed an ok with a seq=1 and a 64-character hex head — you're done with the install. The substrate works on your machine. The rest of this page is what to do with it.

§02 what you just did

What you just did.

The example program at examples/01_wrap_scribe/main.cpp opens a runtime against a fresh SQLite file, pushes the default policy chain, and wraps one inference against a tiny mock scribe callback. On disk, three things appeared next to the binary:

asclepius.db
The SQLite-WAL ledger. One file holds the entire chain — entries, signatures, key registry, drift baselines. Roughly 80 KB after one inference.
asclepius.db.key
The Ed25519 keypair used to sign entries. Mode 0600. Generated on first open if not provided. Move this off the box for production; the runtime will accept it back via Runtime::open(path, KeyStore).
asclepius.db-wal
SQLite's write-ahead log. Transient; rolls into the main file on checkpoint.

Inside asclepius.db there is now exactly one inference.committed entry. The entry records:

  1. the actor — the clinician id the example uses (clinician:dr.demo);

  2. the model id and version, hashed alongside the input and output;

  3. the input hash and output hash — BLAKE2b-256 of the post-policy payload the model saw and produced;

  4. the verdicts of every policy that ran, in order, with their decisions (allow / modify / block);

  5. the previous-entry hash — zero, because this is the first entry — so the next entry will chain to this one.

Every byte of that record is signed with the keypair in asclepius.db.key. The bytes carry the proof. Hand the file to a verifier and they can reconstruct the chain without trusting you.

§03 open it up

Open it up.

The asclepius CLI binary built alongside the example lives at ./build/apps/cli/asclepius. Pin it to your path or invoke directly. Four subcommands cover ninety percent of what you'll do day-to-day. 

# integrity-check the whole chain
$ ./build/apps/cli/asclepius ledger verify ./asclepius.db
  OK · 1 entries · head=8e510bf4...c20e · key=a1f274e6

That command walked the chain, recomputed each entry's hash, re-checked each Ed25519 signature, and confirmed the prev-pointer of every entry matches the hash of the one before. It is the thing the regulator will run. 

# dump the last 5 entries as JSON

§04 wrap your own model

Wrap your own model.

The wrap-scribe example is twenty lines without comments. Replace the mock callback with your own and you have an audited inference. The pattern is always the same — open, push, begin, run, commit. 

#include <asclepius/asclepius.hpp>
using namespace asclepius;

int main() {
    // 1. open the runtime against a SQLite file (created on first open)
    auto rt = Runtime::open("./asclepius.db").value();

    // 2. push policies — order matters; first block wins
    rt.policies().push(make_phi_scrubber());
    rt.policies().push(make_length_limit(8000, 8000));

    // 3. begin one inference, scoped to a clinician + patient + purpose
    auto inf = rt.begin_inference({
        .model     = ModelId{"my-scribe", "v1"},
        .actor     = ActorId::clinician("dr.smith"),
        .patient   = PatientId::pseudonymous("p_9f1a"),
        .encounter = EncounterId::make(),
        .purpose   = Purpose::ambient_documentation,
    }).value();

    // 4. run — the model_call lambda is your model. Anything callable.
    auto model_call = [](std::string_view in) -> std::string {
        // in production: call your transformer, your gRPC stub,
        // your scribe vendor's HTTP endpoint. Whatever returns text.
        return "SOAP: subjective ...\nobjective ...\nassessment ...\nplan ...";
    };

    auto out = inf.run("chief complaint: chest pain, 2h", model_call).value();

    // 5. commit — appends one signed inference.committed entry
    inf.commit().value();

    std::cout << "ok · seq=" << inf.seq().value()
              << " · output=" << out << "\n";
}

Five calls, in order. open gives you a runtime; policies().push wires the input/output filter chain; begin_inference creates a scoped handle that already knows the clinician, patient, and purpose; inf.run sends the input through the input policies, invokes your model callback, sends the output through the output policies, and stages the entry; inf.commit signs and appends.

If a policy blocks, run returns an error and no inference.committed entry is appended — but an inference.aborted entry is, so the refusal itself is auditable. The chain records what didn't happen as well as what did.

compose Need a timeout? inf.run_with_timeout(in, model, 2000ms). Need cancellation? inf.run_cancellable(in, model, token). Need consent? Issue a token in rt.consent() for the patient and purpose; the begin_inference call will refuse without one. The /api page lists every overload.
