How decisions get made.

Asclepius is open-source under Apache 2.0. There is no benevolent dictator; decisions about what merges, what ships, and what breaks are made by maintainers through a small, predictable set of rules. This page is the contract.

Three tiers of change

Every contribution falls into one of three tiers, with a different bar for merging.

tier 01

Routine

Bug fixes with regression tests, doc improvements, performance work that doesn't change behavior. Lazy consensus: any maintainer can merge after 24 hours without an objection. CI must pass.

tier 02

Shape changes

New event types, new policy interfaces, new bundle fields, public-API additions. Require an RFC in rfc/NNNN-title.md, a 7-day comment window, and approval from two maintainers.

tier 03

Breaking changes

Anything that requires asclepius migrate. Require an RFC, a 14-day comment window, a public vote on the mailing list, and a supermajority of active maintainers (≥ 2/3).

Roles

The role definitions are deliberately thin; we add roles only when a tension forces it.

contributor Anyone who has opened a PR, an issue, an RFC, or a bug report. No barriers; no approval needed.
committer Has merge rights to main. Earned by 5+ merged substantive PRs and a maintainer's nomination. Held by ~7 people at any given time.
maintainer Committer who has been active for ≥ 6 months and is responsible for one of: runtime, bindings, site, conformance battery. Sets policy in their domain. ~3 people.
security responder Maintainer who handles security@asclepius.health. At least 2 at any given time. Rotates monthly.

RFC process

An RFC is a Markdown file in rfc/NNNN-title.md. The next available NNNN is whatever's next; collisions resolve by re-numbering at merge time. Required sections:

  • Problem — what is broken without this
  • Non-goals — what this RFC will not address
  • Design — the proposed change, in enough detail to implement
  • Compatibility — does this break v1.x; if so, why is the break worth it
  • Drawbacks — honest accounting of what gets worse
  • Alternatives — at least two; explain why the chosen path won
  • Open questions — what we still don't know

RFCs merge when two maintainers approve. Merging an RFC commits us to discuss; the implementation PR carries the RFC number in its subject line (rfc/0042). An RFC may be withdrawn by its author at any point before implementation lands.

Disagreement

If reasonable contributors disagree and lazy consensus stalls, the path is:

  • One week of focused discussion in the PR thread.
  • If unresolved: a maintainer call (asynchronous, in the PR), 5 working days.
  • If still unresolved: a vote on the public mailing list, simple majority of active maintainers.
  • Anything that survives a vote ships, with a written summary in the PR explaining the trade-off.

We have not yet had to escalate past step one. We document the rule because we want to keep that record clean.

Releases

Releases happen on a readiness basis, not on a calendar. The release manager rotates monthly among the maintainers; their job is to cut the tag, write the changelog entry, and run the conformance battery against the release candidate. A release fails CI if the conformance battery fails on any supported platform — full stop.

Patch releases (0.x.y) need 1 maintainer approval. Minor releases need 2. Major releases need 2 + an RFC describing the migration path.

Trademarks & the brand

The name "Asclepius" and the seal/chain-head mark are not covered by Apache 2.0. Use the press kit at /press for what's permitted; everything not covered there should be raised as an issue on GitHub first. The substrate code itself can be vendored, forked, and rebranded under Apache 2.0 with no permission needed.

Code of conduct

Adapted from the Contributor Covenant and lightly opinionated. The summary: assume good faith, prefer precise critique over personal critique, remember that everything that happens in this project will run next to a patient. Detailed policy + reporting flow in /contributing.

Why this is short

A governance document that is longer than the codebase governs nothing. The above fits on one printed page. If a situation arises that this page does not cover, we will write down what we decided to do and add it here. The git log of this file is the canonical history.
