crypto toolbox · interactive · in-browser

The same primitives, in your tab.

Three small tools for building intuition about the cryptography Asclepius runs on. Everything below executes locally in your browser using Web Crypto — no network calls, no telemetry, nothing leaves the page. The C++ runtime uses BLAKE2b-256 + Ed25519 via libsodium; this page demonstrates the shape of those operations on data you supply.

§ 01 · hash visualizer

Same input, three fingerprints.

Hash any text and watch SHA-256 / SHA-384 / SHA-512 update live as you type. Web Crypto exposes all three; the runtime's BLAKE2b-256 isn't shown because Web Crypto doesn't ship it — for that, the CLI: echo -n "x" | asclepius hash.

SHA-256—32 bytes
SHA-384—48 bytes
SHA-512—64 bytes
why three? Asclepius ledgers store fixed-length payload + entry hashes. The hash is content-addressed: two equal-shaped bodies produce byte-equal digests across machines, runtimes, and decades. Change one bit of the input — watch every digest change.
§ 02 · ed25519 sign & verify

One keypair, one signature, irrefutable.

Generate a fresh Ed25519 keypair (or ECDSA-P256 fallback on browsers without Ed25519), sign a message, verify it. Then mutate one byte and watch verification fail.

what just happened The signature is 64 bytes (Ed25519) or ~70 (ECDSA-P256), and it covers the exact bytes of your message. Mutate one character — verify fails. The C++ runtime carries the same shape: every entry's signature is over seq | ts | prev | payload | actor | event | tenant | body, so flipping any of those reduces verification to "no".
§ 03 · chain-forge demo

An attacker without the secret key cannot succeed.

Build a 5-row chain (each row's prev_hash equals the previous row's hash). Click forge to attempt an attack: modify row 3's body. Without the original signing key, the signature no longer covers the new bytes — verify catches it at exactly that row.
