Send a PR. Or read the chain first.

Asclepius is open-source under Apache 2.0. The codebase is small, opinionated, and deliberately under-abstracted. Anything you ship here ends up running on a clinician's desk. Read the threat model first, then the architecture, then the spec — in that order.

01

Read first

Architecture · spec · threat model. Most rejections happen because a PR misunderstood one of these.

02

RFC if it's a shape

New event type, new policy interface, new bundle field — open a thread in rfc/ first.

03

Branch & build

cmake --preset=dev && ctest -L unit -j. Sanitisers ship on by default.

04

PR with a why

Subject line is the why, not the what. Tests required for non-doc changes.

05

Sign-off

DCO required: git commit -s. Two-maintainer review before merge.

Code style — no surprises

We follow the .clang-format in the repo (LLVM-derived, 4-space indent, columns soft-wrapped at 100). One non-obvious convention: prefer composition over inheritance; the only abstract class in the runtime is IPolicy, and that is a deliberate exception.

Headers are kept lean — #include only what each file needs. nlohmann/json_fwd.hpp in headers, nlohmann/json.hpp in TUs. No using namespace in headers. noexcept is opt-in: claim it only when the body honestly is. 

$ cmake --preset=dev
$ cmake --build --preset=dev -j
$ ctest --preset=dev -j -L unit
$ clang-format -i $(git diff --name-only --diff-filter=AM)

What we accept, what we don't

YESbug fix with a regression test that fails before, passes after
YESnew built-in policy implementing IPolicy with PHI-safe defaults
YESnew event type, gated behind an RFC + a SPEC.md update
YEStighter verify() — refusing earlier on a structural mismatch
YESperf improvement with bench numbers in the PR description
NOformat breakage without a asclepius migrate script
NOswitching to a non-libsodium primitive without a threat-model update
NO"clean-up" PRs that rename or move files without a justification
NOintroducing a runtime dependency that calls home (no analytics, ever)
NOadding a feature flag for something we won't enable in a release

RFC process

Anything that changes a public type, a wire shape, or a ledger event needs an RFC. Open a Markdown file under rfc/NNNN-title.md with these sections: Problem · Non-goals · Design · Compatibility · Drawbacks · Alternatives · Open questions. RFCs are merged when two maintainers approve and there is a one-week comment window.

An RFC merge does not commit us to ship; it commits us to discuss. The implementation PR carries the RFC number in its subject (rfc/0042).

Security disclosures

Found a vulnerability? Do not open a public issue. Email security@asclepius.health with details and a PoC if available. We acknowledge within 48 hours, fix within 14 days for high-severity issues, coordinate disclosure, and credit you on the security advisory. See /security for our PGP key, scope, and SLA.

Code of conduct

Adapted from the Contributor Covenant. The summary: assume good faith, prefer precise critique over personal critique, and remember that everything that happens in this project will, eventually, run next to a patient. Disrespect of either contributors or the patients downstream of our code is unacceptable.

Where to start

Issues labelled good-first-issue are scoped to one or two days and have an explicit acceptance test. help-wanted are larger and benefit from a comment before you begin. The friendliest path in is improving the conformance test corpus — public, benchmarked, regulator-relevant, and pure-data.
