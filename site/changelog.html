What ships, and when.

Asclepius follows semver. The runtime, on-disk format, and on-wire schema are stable across the v1.x line; pre-1.0 releases may break compatibility, but every break is annotated below and migrated by a one-shot script.

§ roadmap

Shipped now

The substrate as it stands today: a working runtime, a working ledger, a working verifier.

  • Signed Merkle ledger
  • Composable policy chain
  • Drift monitor (PSI / KS / EMD)
  • Consent registry
  • Evidence bundle export + verify
  • Python bindings (pybind11)
  • CLI: ledger / evidence / drift

In flight (2026 Q3)

Active development. PRs welcome.

  • gRPC sidecar (asclepius-svc)
  • Postgres ledger backend
  • HSM-backed KeyStore (PKCS#11)
  • Rust bindings (cxx)
  • Encrypted body-store (opt-in PHI retention)
  • Streaming inverse-RPC for model callbacks

Considered (post-1.0)

Frequently requested; designs in flight, not yet committed to a release.

  • Policy DSL + asclepius policy lint
  • Multi-key ledger (rotation in-place)
  • Notarization adapters (Sigstore, OpenTimestamps)
  • Embedding-drift on top of scalar drift
  • Post-quantum signer (when standardized)
  • Managed evidence service (commercial offering)

§ compatibility guarantees

Patch (0.1.x): bug fixes only. Binary-compatible. No format changes.

Minor (0.x.0): additive changes only. New event types, new fields, new policies. Old verifiers ignore unknown fields.

Major (x.0.0): incompatible changes are allowed. A migration script ships with every major bump (asclepius migrate vN-1_to_vN). Bundles emitted by version N continue to verify with the same major version forever.

v1.0 onward: the on-disk ledger format and on-wire sidecar schema are frozen. Removing or renaming a field is a v2 break and requires a new schema package (asclepius.v2).
