accessibility statement · 2026-04-26

A trust substrate has to be reachable.

We target WCAG 2.2 Level AA for every public-facing page on asclepius.health, including the interactive playground. This page documents what we've verified, what we haven't, what we know is broken, and how to tell us about something we missed.

Current conformance

Audited internally on 2026-04-26 against the WCAG 2.2 AA criteria. Issues were re-verified with NVDA 2024.4 (Windows + Firefox) and VoiceOver (macOS Sonoma + Safari) plus axe-core 4.10 in CI.

1.1Text alternatives — images use alt or aria-label; decorative SVGs are aria-hidden.pass
1.3Adaptable content — semantic HTML, headings hierarchical, no pure-style table elements.pass
1.4Distinguishable — 4.5:1 contrast minimum, no information by color alone, supports user font-size up to 200%.pass
2.1Keyboard — every interactive element reachable; ? opens help, ⌘K opens palette.pass
2.2Enough time — no auto-advancing carousels, no time-limited content.pass
2.3Seizures — no flashing > 3 Hz; prefers-reduced-motion disables every animation.pass
2.4Navigable — skip-link on every page, focus visible, page titles unique, breadcrumbs implicit via <nav> landmarks.pass
2.5Input modalities — pointer + keyboard equivalent, no drag-only operations.pass
3.1Readable — language declared, glossary at /glossary, no jargon without definition on first use.pass
3.2Predictable — focus does not navigate, theme toggle is opt-in, no surprise pop-ups.pass
3.3Input assistance — all forms have labels; the playground textarea announces its purpose to AT.pass
4.1Compatible — valid HTML, ARIA used only when native semantics insufficient, no name+role+value gaps.pass

What we did, mechanically

  • Skip-to-content link on every page; visible only on focus.
  • Visible focus rings via :focus-visible, never removed.
  • Semantic landmarks — one <header>, one <main>, one <footer> per page; <nav aria-label> on every menu.
  • Heading hierarchy validated by check_html.py in CI — no h1→h3 jumps.
  • Reduced-motion respected — reveal-on-scroll, chain-pulse, type-on-scroll all disabled when prefers-reduced-motion: reduce.
  • Color contrast — text-on-background pairs verified at AA in both themes; decorative italic accents (purple on dark, deep purple on parchment) sit at 7:1+.
  • ARIA on dynamic regions — the playground ledger uses aria-live="polite"; modals declare role="dialog" + aria-modal="true".
  • Keyboard chord navigation — see ? for the full list.

Known issues — being worked on

We keep this list small and current. If you don't see your issue here, see How to report below.

Drift histogram has no text alternative. 1.1.1 in flight
The bar chart on /demo updates as you wrap inferences but currently exposes only an aria-label of 20-bin confidence histogram. We're adding a live-updating aria-live textual summary (PSI 0.34, severity moderate, 8 of 20 bins) that screen readers can follow without parsing the visual. Target: v0.4.0.
Cmd-K palette traps tab focus only after the first arrow-key press. 2.4.3 in flight
The first focused element is the search input; Tab escapes the dialog before the focus-trap engages. We're moving the trap to focusin on document.activeElement instead of waiting for a keydown event. Target: 2026-05.
Demo's "wrap inference" announces success but not policy verdicts. 4.1.3 in flight
Each verdict (allow/modify/block) updates a visual chip but only the final status reaches the aria-live region. Sighted users see five state changes; screen-reader users hear one. We're consolidating verdicts into a single coherent announcement (PHI scrubber modified · schema validator allowed · committed seq 0001). Target: v0.4.0.

Out of scope

We do not currently target WCAG 2.2 AAA — some criteria (e.g., 1.4.6 Contrast Enhanced at 7:1 for all body text, 2.4.9 Link Purpose by link text alone in all contexts) would compromise the editorial register the site is built around. We document what we don't do and choose accordingly. If your assistive workflow needs an AAA-conformant view, the plain-Markdown source is the substrate of every page and is unconditionally readable.

How to report

Email a11y@asclepius.health with the URL, a description of the issue, your assistive technology and version, and (if you can) a screenshot or recording. We acknowledge within 3 working days. Critical (Level A) issues are treated as security-class incidents — fixed within 14 days, with the fix surfaced on /status.

You can also open a public issue on GitHub with the accessibility label.
