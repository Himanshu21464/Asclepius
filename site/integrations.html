§01 · seven recipes · twelve lines each · vendor-neutral

Wrap anything in twelve lines.

Asclepius is deliberately not a model. It is the substrate that wraps one. Below are seven concrete recipes — an ambient-scribe vendor, an in-house imaging model, a tool-using agent, a FHIR sidecar, a batch evaluation harness, an Azure-OpenAI lab-summary demo (Microsoft's Asclepius), and an open clinical LLM (starmpcc/Asclepius) — with C++ and Python side-by-side, and the ledger row each one produces. Copy, swap the model, ship.

§01 · scribe vendor · 12 lines

An ambient-scribe vendor.

The vendor exposes scribe.transcribe(audio) → SOAP. You wrap it. The vendor never sees identifiers; you never lose the audit trail. The vendor's API does not change a byte. 

// 12-line scribe wrap — input goes through phi+schema+length policies,
// output is signed and committed to the Merkle ledger.
#include <asclepius/asclepius.hpp>

asc::Runtime rt = asc::Runtime::open("./asclepius.db").value();
rt.policies().push(asc::policy::PhiScrub::v4());
rt.policies().push(asc::policy::SchemaTranscript::v2());
rt.policies().push(asc::policy::Length::v1(8192));

auto inf = rt.begin_inference({.actor="clin:np_marisol_h",
                                .patient=patient_id,
                                .model={"scribe-v3", weights_sha256},
                                .purpose=asc::Purpose::Scribe});
auto safe_in  = inf.policy_chain(transcript);          // PHI-scrubbed
auto soap     = vendor::scribe_transcribe(safe_in);     // the model boundary
inf.commit(soap);                                       // signs + appends + drift
import asclepius as asc, vendor

rt = asc.Runtime.open("./asclepius.db")
rt.policies.push(asc.policy.phi_scrub_v4())
rt.policies.push(asc.policy.schema_transcript_v2())
rt.policies.push(asc.policy.length_v1(8192))

with rt.begin_inference(actor="clin:np_marisol_h",
                        patient=patient_id,
                        model=("scribe-v3", weights_sha256),
                        purpose=asc.Purpose.Scribe) as inf:
    safe_in = inf.policy_chain(transcript)
    soap    = vendor.scribe_transcribe(safe_in)
    inf.commit(soap)

resulting ledger row

{ "seq": 874, "kind": "inference.completed",
  "model": { "id": "scribe-v3", "weights_sha256": "6a0f…e2" },
  "policies": ["phi.v4", "schema.transcript.v2", "length.v1"],

What you keep. The vendor's IP. Their API. Their model. What you gain: bit-identical input/output hashes the EHR can replay against, and a chain whose root the operator publishes weekly. The audit is not parallel work — it is the return path.

§02 · in-house imaging · 14 lines

An in-house imaging model.

Your team trained a chest-CT classifier. It runs in your Kubernetes cluster. You want every suggestion testifiable. The substrate doesn't care that the model is yours — provenance is provenance. 

asc::Runtime rt = asc::Runtime::open("./asclepius.db").value();
rt.policies().push(asc::policy::DicomStrip::v1());
rt.policies().push(asc::policy::SchemaImaging::v1());
rt.policies().push(asc::policy::ConfidenceFloor::v1(0.6));
rt.policies().push(asc::policy::ActionFilter::advisory_only());

auto inf = rt.begin_inference({.actor="svc:imaging-fm",
                                .patient=patient_id,
                                .model={"chest-ct-v2", weights_sha256},
                                .purpose=asc::Purpose::Diagnostic});
auto safe_dicom = inf.policy_chain(dicom_series);
auto finding    = our_model::infer(safe_dicom);            // {finding, conf, ROIs}
auto [seq, sig] = inf.commit(finding);                     // signed, ledgered
publish_to_pacs(finding, seq, sig);                          // radiologist sees it
import asclepius as asc, our_model

rt = asc.Runtime.open("./asclepius.db")
rt.policies.push(asc.policy.dicom_strip_v1())
rt.policies.push(asc.policy.schema_imaging_v1())
rt.policies.push(asc.policy.confidence_floor_v1(0.6))
rt.policies.push(asc.policy.action_filter_advisory())

with rt.begin_inference(actor="svc:imaging-fm",
                        patient=patient_id,
                        model=("chest-ct-v2", weights_sha256),
                        purpose=asc.Purpose.Diagnostic) as inf:
    safe = inf.policy_chain(dicom_series)
    out  = our_model.infer(safe)
    seq, sig = inf.commit(out)
    publish_to_pacs(out, seq, sig)

§03 · tool-using agent · 16 lines

A tool-using agent.

The agent proposes a plan. The substrate enforces the gate: low-blast-radius actions auto-commit; high-blast-radius actions stage and wait for a signed sign-off. Two keys, two rows, one chain. 

asc::Runtime rt = asc::Runtime::open("./asclepius.db").value();
rt.policies().push(asc::policy::PhiScrub::v4());
rt.policies().push(asc::policy::SchemaActions::v1());
rt.policies().push(asc::policy::ActionFilter::from_yaml("blast_radius.yml"));

auto inf = rt.begin_inference({.actor="agent:triage-1",
                                .patient=patient_id,
                                .model={"triage-agent-1", weights_sha256},
                                .purpose=asc::Purpose::Agentic});
auto safe = inf.policy_chain(chart_context);
auto plan = agent::propose(safe);                          // vector<Action>
auto verdict = inf.dispatch(plan);                         // staged or committed

for (auto& staged : verdict.staged) {
    auto read_back = attending::confirm(staged);
    rt.cosign(staged, read_back, "clin:dr_yamamoto");     // human key, second sig
}
rt = asc.Runtime.open("./asclepius.db")
rt.policies.push(asc.policy.phi_scrub_v4())
rt.policies.push(asc.policy.schema_actions_v1())
rt.policies.push(asc.policy.action_filter_from_yaml("blast_radius.yml"))

with rt.begin_inference(actor="agent:triage-1",
                        patient=patient_id,
                        model=("triage-agent-1", weights_sha256),
                        purpose=asc.Purpose.Agentic) as inf:
    safe    = inf.policy_chain(chart_context)
    plan    = agent.propose(safe)
    verdict = inf.dispatch(plan)

    for staged in verdict.staged:
        read_back = attending.confirm(staged)
        rt.cosign(staged, read_back, signer="clin:dr_yamamoto")

two-row outcome (auto-commit + staged + co-signed)

[ { "seq":3331, "kind":"agent.proposed",     "actions":["order.cxr","order.heparin.continuous"], … },
  { "seq":3332, "kind":"action.committed",   "action":"order.cxr",                "refers_to":"seq:3331" },
  { "seq":3333, "kind":"action.staged",      "action":"order.heparin.continuous", "awaiting":"clin:attending" },
  { "seq":3340, "kind":"action.committed",   "action":"order.heparin.continuous", "signed_by":"clin:dr_yamamoto" } ]

Why dispatch, not commit? dispatch consults the action filter and routes each action to its right gate. The agent never knows whether an action auto-committed or staged — that is the operator's policy decision, not the model's.

§04 · FHIR sidecar · 10 lines

A FHIR sidecar.

The EHR speaks FHIR. The model speaks tensors. The sidecar speaks both — it accepts a Bundle, runs an inference through Asclepius, and returns a DocumentReference whose identifier.value is the ledger seq. 

// asclepius-svc handler — POST /v1/wrap
asc::svc::handle("/v1/wrap", [&](asc::svc::Request& req) {
    asc::interop::FhirBundle b = asc::interop::parse(req.body());
    auto inf = rt.begin_inference(asc::interop::context_from(b));
    auto safe = inf.policy_chain(b.payload());
    auto out  = forward_to_model(safe);
    auto [seq, sig] = inf.commit(out);
    return asc::svc::ok(asc::interop::doc_ref(out, seq, sig));
});
from asclepius.svc import handle, ok
from asclepius.interop import parse, context_from, doc_ref

@handle("/v1/wrap")
def wrap(req):
    bundle = parse(req.body)
    with rt.begin_inference(context_from(bundle)) as inf:
        out = forward_to_model(inf.policy_chain(bundle.payload))
        seq, sig = inf.commit(out)
        return ok(doc_ref(out, seq, sig))

What you write. Five lines of glue. What you get. A FHIR-native endpoint whose responses bind every clinical artefact to a verifiable ledger row. The EHR does not need to know Asclepius exists; it sees a normal DocumentReference with an identifier it can audit later.

§05 · batch evaluation · 11 lines

A batch evaluation harness.

Before a model goes live, you run it against a held-out evaluation set and compute drift, calibration, and policy-violation rate. Every prediction is ledgered with purpose: qa — so the eval is itself a signed artefact you can hand to a regulator. Validation is provenance. 

asc::EvaluationHarness eval(rt, asc::Purpose::QA);

for (auto& sample : test_set) {
    auto inf  = eval.begin(sample.model_id, sample.weights_sha256);
    auto pred = our_model::infer(inf.policy_chain(sample.input));
    eval.observe(inf, pred, sample.gold);                  // updates calibration + drift
}

auto report = eval.finalise();                              // signed report
asc::tar::write_bundle("qa-2026-04-26.tar", report, eval.entries());
eval = asc.EvaluationHarness(rt, purpose=asc.Purpose.QA)

for sample in test_set:
    with eval.begin(sample.model_id, sample.weights_sha256) as inf:
        pred = our_model.infer(inf.policy_chain(sample.input))
        eval.observe(inf, pred, sample.gold)

report = eval.finalise()
asc.tar.write_bundle("qa-2026-04-26.tar", report, eval.entries())

resulting bundle (USTAR tar — opens with stock tar)

qa-2026-04-26.tar
├── manifest.json           BLAKE2b of every file inside
├── ledger.jsonl            every prediction as a signed entry
├── drift.json              PSI/KS/EMD per feature, with thresholds
├── calibration.json        reliability curve, ECE, MCE
├── policy_violations.jsonl rows that failed any policy
└── pubkey.ed25519          the verifier needs this only — nothing else
§06 · same-name wrap · Microsoft Asclepius · 13 lines

Microsoft Asclepius.

The microsoft/asclepius repo is a sample app: Azure OpenAI generating patient-friendly summaries of lab results, fed by a Python Function App over Semantic Kernel. It is a clinical-AI tool — exactly the kind we are designed to wrap. Different layer, same name; see ADR-012. Wrapping it is the boring kind of boring. 

// 13 lines — wrap the MS sample's Function App behind the substrate.
asc::Runtime rt = asc::Runtime::open("./asclepius.db").value();
rt.policies().push(asc::policy::PhiScrub::v4());
rt.policies().push(asc::policy::SchemaLabSummary::v1());
rt.policies().push(asc::policy::Length::v1(4096));

auto inf = rt.begin_inference({.actor="clin:dr_park",
                                .patient=patient_id,
                                .model={"ms-asclepius:semantic-kernel", fn_app_sha256},
                                .purpose=asc::Purpose::PatientFacing});
auto safe_ctx = inf.policy_chain(json{ {"labs", labs}, {"chief", chief}, {"hx", history} });
auto summary = ms_asclepius::generate_summary(safe_ctx);  // HTTPs to Azure Function
inf.commit(summary);                                       // signs + appends + drift
import asclepius as asc, ms_asclepius

rt = asc.Runtime.open("./asclepius.db")
rt.policies.push(asc.policy.phi_scrub_v4())
rt.policies.push(asc.policy.schema_lab_summary_v1())
rt.policies.push(asc.policy.length_v1(4096))

with rt.begin_inference(actor="clin:dr_park",
                        patient=patient_id,
                        model=("ms-asclepius:semantic-kernel", fn_app_sha256),
                        purpose=asc.Purpose.PatientFacing) as inf:

§07 · same-name wrap · starmpcc/Asclepius · 12 lines

An open clinical LLM (starmpcc/Asclepius).

starmpcc/Asclepius is a clinical LLM (7B/13B/Llama3-8B/Mistral) trained on synthetic discharge summaries. Weights live on Hugging Face; the model takes (note, instruction) and returns a structured response. We wrap it the same way we wrap anything else. 

// 12 lines — model is loaded over Python boundary; runtime is C++.
asc::Runtime rt = asc::Runtime::open("./asclepius.db").value();
rt.policies().push(asc::policy::PhiScrub::v4());
rt.policies().push(asc::policy::SchemaDischargeQA::v1());
rt.policies().push(asc::policy::ClinicalActionFilter::from_yaml("actions.yml"));

auto inf = rt.begin_inference({.actor="clin:dr_kwon",
                                .patient=patient_id,
                                .model={"starmpcc/Asclepius-Llama3-8B", weights_sha256},
                                .purpose=asc::Purpose::DecisionSupport});
auto safe = inf.policy_chain({.note=discharge_note, .instr=instruction});
auto answer = py::call("asclepius_llama3", safe);
inf.commit(answer);
import asclepius as asc
from transformers import AutoModelForCausalLM, AutoTokenizer

tok = AutoTokenizer.from_pretrained("starmpcc/Asclepius-Llama3-8B")
mdl = AutoModelForCausalLM.from_pretrained("starmpcc/Asclepius-Llama3-8B")

rt = asc.Runtime.open("./asclepius.db")
rt.policies.push(asc.policy.phi_scrub_v4())
rt.policies.push(asc.policy.schema_discharge_qa_v1())
rt.policies.push(asc.policy.clinical_action_filter_from_yaml("actions.yml"))

with rt.begin_inference(actor="clin:dr_kwon",
                        patient=patient_id,
                        model=("starmpcc/Asclepius-Llama3-8B", weights_sha256),
                        purpose=asc.Purpose.DecisionSupport) as inf:
    safe   = inf.policy_chain({"note": discharge_note, "instr": instruction})
    answer = mdl.generate(**tok(asc.fmt.starmpcc(safe), return_tensors="pt"))
    inf.commit(tok.decode(answer[0], skip_special_tokens=True))

§08 · ABDM (India) · kernel primitives r90 · end-to-end demo at examples/07_india_profile/

ABDM (India).

ABDM (India) — kernel primitives ConsentArtefact, FamilyGraph, EmergencyOverride are in the kernel today and exercised by a runnable end-to-end demo at examples/07_india_profile/. The Gateway adapter, the HIP/HIU plumbing, and the ABHA discovery flow remain an operator concern — they ship as an example, not as kernel code. Wire shape lives in the kernel; transport lives above it. 

// kernel-side: token in registry → outbound ABDM consent artefact.
asc::ConsentArtefact a = asc::artefact_from_token(
    token,                            // in-memory ConsentToken
    "hiu:apollo-blr",                  // requester (HIU)
    "hip:fortis-tnk",                  // fetcher (HIP)
    "abdm:abc-123-def");                // ABDM artefact id
std::string wire = asc::to_abdm_json(a); // → adapter sends to Gateway

// proxy consent: an adult child for an elder parent.
asc::FamilyGraph fg;
fg.record_relation(adult_child, elder_parent,
                   asc::FamilyRelation::adult_child_for_elder_parent);

// break-glass: clinician on an unconscious patient, 72h backfill window.
asc::EmergencyOverride eo;
auto tok = eo.activate(actor, patient, "RTA, GCS 7").value();
// later, within 72h:
eo.backfill(tok.token_id, "signed-clinical-note:0x4a");
import asclepius as asc

# token in registry → outbound ABDM artefact
artefact = asc.artefact_from_token(token,
                                   requester_id="hiu:apollo-blr",
                                   fetcher_id="hip:fortis-tnk",
                                   artefact_id="abdm:abc-123-def")
wire = asc.to_abdm_json(artefact)

# proxy consent: adult child for elder parent
fg = asc.FamilyGraph()
fg.record_relation(adult_child, elder_parent,
                   asc.FamilyRelation.ADULT_CHILD_FOR_ELDER_PARENT)

# break-glass with 72h backfill window
eo = asc.EmergencyOverride()
tok = eo.activate(actor, patient, reason="RTA, GCS 7")
eo.backfill(tok.token_id, evidence_id="signed-clinical-note:0x4a")

Why kernel, why not adapter? The wire shape, the proxy graph, and the deferred-consent timer are contracts, not transports. Pushing them into per-deployment adapter code would have every Indian operator re-deriving the same JSON schema and the same 72-hour clock — and the audit chain would not reconstruct family-proxy consent from the ledger alone. The transport (Gateway HTTP, ABHA discovery, retry policy) ships in examples/abha/; the kernel just records and looks up. See /india and ADR-013.

The pattern is always the same.

Push the policy chain. Open an inference handle. Run the chain on the input. Hand the safe input to the model. Commit the output. The substrate handles hashing, signing, drift, consent, and bundle export — none of which the model ever sees, and none of which you implement.

Adding a new vendor or model is, by design, the boring part. The interesting work — the cryptography, the ledger consistency, the drift maths, the bundle format — is the substrate's job, not yours.
